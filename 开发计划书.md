## 🔍 系统迭代机制

> 每个模块不是"设计→实现→完"，而是**三轮迭代**逐步打磨。利用已有设计文档做逐段审查，低成本发现矛盾。

### 三轮迭代模型

```
第 1 轮：设计审查（读文档，找矛盾）
  ├── 逻辑自相矛盾？（状态机/数值/边界条件）
  ├── 遗漏子系统？（只列文件名，无设计段）
  ├── 职责边界不清？（两个模块抢同一件事）
  └── 产出：问题列表 + 文档修复

第 2 轮：接入审查（查联动，对齐接口）
  ├── 事件是否对齐？（A 发的 B 能收到？字段名一致？）
  ├── 数据是否传递？（IDamageable/TakeDamage/ThreatReport 全链路）
  ├── 时序是否正确？（Awake→OnEnable→Start 顺序依赖）
  └── 产出：跨系统接口修正 + 对接测试

第 3 轮：边缘/性能（压测，找瓶颈）
  ├── 极端情况（全死了/全活着/空场景/NPC×100）
  ├── 性能瓶颈（GC 分配/每帧 alloc/集合遍历）
  ├── 边界条件（NavMesh 丢失/能量耗尽/引用 null）
  └── 产出：压测报告 + 优化补丁
```

## 🔄 阶段 2：世界骨架（代码 + 数据）

> 核心原则：**按依赖层级开发，不跳级**。
> L1 → L2 → L3 → L4，每一层为上一层提供材料。

### 层级总图

```
L1 地基 ─── 任何系统都依赖它
  ├─ ItemData (SO) ✅     物品是一切的基础

  ├─ 移动/摄像机 ✅       玩家操作基础
  └─ 碰撞/物理 ✅         世界交互基础

  ├─ 战斗系统 ✅          Phase1~6全完成（武器数据+装备+模型+瞄准+射击+伤害）
  ├─ 建造系统 ✅          Phase1完成（Ghost预览+放置+拆除+BuildMenuUI+7默认资产）
```

##### 功能详情

**PlacedItem 扩展**:
- 新增 `rotated` 字段(bool)，标记物品是否旋转90°

**坐标检测拖拽 (DragDropManager)**:
- `CellRegion` 结构体: 记录每个格子的 RectTransform + 所属容器 + 网格坐标 + 是否武器槽
- `FindCellAtPosition(screenPos)`: 反向遍历已注册格子，用 `RectTransformUtility.RectangleContainsScreenPoint` 判定落点

- 按住未拖时: 原地旋转 (`RotateInPlace`)，在源容器内更新位置，用 `IsSpaceFreeFor` 排除自身格子
- 拖拽中: toggle `_dragRotated` 标记，落点时校验
- 已选中时: 旋转选中物品 (`RotateSelectedItem`)
- 旋转公式: `(rx, ry) → (ry, rx)`，原点格子不动，矩形绕原点旋转90°

**选中系统**:
- 点击物品立刻显示白框，长按持续存在
- 白框实现: 在 overlay 上添加 4 条白色 Image (SelTop/SelBot/SelLeft/SelRight)，不触发 UI 重建
- `RefreshSelectionBorder()`: UI 重建后通过已注册格子查找 overlay 恢复白框

**放置逻辑 (TryMoveToContainer)**:
- 先检测是否为武器槽自动装备
- 同容器旋转检测用 `IsSpaceFreeFor`（排除自身格子）
- 不同容器直接 `IsSpaceFree`
- 放不下时回退到原位，弹出 Toast

##### 电网拓扑：两级连接

```
发电端 ──[电缆]──→ 用电终端 ──[圆形范围]──→ 设备自动供电
                     │
                     └──[电缆]──→ 另一个终端 ──[圆形范围]──→ 更多设备
```

- **发电端 → 终端**：需要拉一条电缆（有建造感，但不繁琐）
- **终端 → 终端**：可以串联，全电网共享一个总功率池
- **终端 → 设备**：圆形范围内自动供电，无需逐设备接线
- **电网定义**：所有用电缆连在一起的终端 + 它们接入的发电端 = 一个电网
- **功率更新**：任何终端接入新发电端 → 整个电网的功率池自动更新

##### 设备双模驱动：烧煤 vs 电力

> 核心理念：**不是禁止烧煤，而是烧到你主动想接电。** 用代价引导，不强制。

**前期 — 只有烧煤，养成习惯**

| 设备 | 烧煤 | 电力 |
|------|------|:--:|
| 篝火 | 木柴燃料，每次烹饪1个 | ✗ |
| 熔炉 | 煤炭/木炭，每炉1个 | ✗ |

**中期 — 电力来了，烧煤开始显露出问题**

| 设备 | 烧煤模式 | 电力模式 |
|------|------|------|
| 高效蒸馏装置 | 木炭×2/轮，产量基准 | 产量×2，速度×2 |
| 半自动化工台 | 煤炭×1/轮，手动投料 | 自动投料，速度×1.5 |
| 半自动制药台 | 木炭×1/轮，**10%报废率** | 零报废 |
| 铸铁火炉 | 木柴/煤炭，60分钟一炉 | 恒温持续供暖 |
| 简易织布机 | ✗ | 手摇→电动，产出×3 |

**后期 — 烧煤烧到你肉痛**

| 设备 | 烧煤模式 | 电力模式 |

| 食品加工厂 | 木炭×2/轮 | 自动，产量×2 |

```csharp
    public List<PowerTerminal> connectedTerminals;

    void RecalculateGrid();           // 终端接入/断开时重新计算全网功率
}

public class PowerConsumer : MonoBehaviour
{
```

- 基础500HP，<30%=重伤(切跟随)，<10%=贴紧玩家(距离1m)

### 4.7.1 AI机器人驾驶系统 ✅（已完成 · 2026-05-25）

> **依赖**: AIBot + AIBotCombat + CameraFollow + InputRouter
> **目标**：玩家可驾驶AI机器人，手动控制一门武器，其余AI自动
> **Git**: 待提交

##### 需求概述

1. **驾驶按钮** — AI面板修理按钮下方新增"驾驶"toggle，bool开关：点一下进入，再点退出
2. **驾驶模式** — 玩家模型隐藏、碰撞体禁用、摄像机切到机器人
3. **武器切换** — 驾驶中按 **Q** 循环三武器（激光→右臂→左臂），当前武器手动控制，其余两个AI自动
4. **AI接管武器按钮** — 旁边加toggle，开启后AI接管全部三把武器，玩家不可开火
5. **E 永远打开 AI 面板** — 驾驶中也不例外，玩家需打开面板点击"驾驶"按钮退出

##### 射击逻辑

- **右臂/左臂武器** — 照搬玩家射击逻辑：鼠标瞄准 → 左键 → 射线检测(Raycast从机器人朝向瞄准点) → 命中僵尸 → 造成伤害。与 `WeaponShooting` 完全相同的射线检测流程
- **激光武器（特殊）** — 左键自动锁敌：`OverlapSphere` 扫描范围内最近僵尸 → 自动锁定 → 发射激光。无需手动瞄准，类似自瞄。调用 `ManualFireLaser()` 而非 `ManualFireAimed()`

全部 7 个步骤已完成。实际开发中 InputRouter 的优先级机制在驾驶场景下不稳定，改为 AIBotPilot.Update() 直接用 `Input.GetKeyDown`/`Input.GetMouseButtonDown` 绕开 InputRouter。

| AI无人机离线 | 仓内普通无人机停靠等待 |
| AIBot死亡 | 仓内+空中所有无人机立即销毁 |

##### AI无人机 = 编队大脑

```
AI无人机 → 独立AI，可单独操控（通过AIBot面板）
     ↓
普通无人机×N → 编队模式，共享指令，不单独操控
```

**为什么不让每架普通无人机单独操控？** 3架普通无人机 + 1架AI无人机 + AIBot本体 = 5个可操控单位，太繁琐。编队模式把普通无人机统一管理，AI无人机才可单独微操。

##### 编队指令

| 编队指令 | 行为 |
|------|------|
| **跟随** | 小半径(5-8m)环绕 AIBot，高度分层错开 |
| **警戒** | 大半径(20-30m)分散巡逻，每人覆盖一个扇形区域 |
| **定点** | 锁定地图某点，编队整体在该点上空巡航 |
| **进攻** | 优先 AIBot 主目标 → 其次各自就近选目标，避免重复锁定 |
| **停火** | 不攻击，只飞不输出 |
| **全员返航** | 全部返回AIBot仓 |

##### 空中空间分离

多架在同一圆上巡航，自动错开相位+不同高度层：

```
N=3 时：
  无人机1: 高度 12m, 相位角 0°
  无人机2: 高度 18m, 相位角 120°
  无人机3: 高度 24m, 相位角 240°

phaseOffset[i] = i × 360° / N
height[i] = 10 + i × 6
```

##### 轻量级目标分配（防扎堆打同一只）

每架普通无人机独立Tick，无需中心调度器：

```
1. 读 AIBot 主目标 → 在攻击锥内? → 锁定（协同集火）
2. 主目标不在范围 → 扫描附近僵尸
3. 过滤掉"已被 ≥1 架友军锁定"的僵尸（防扎堆）
4. 选最近未被锁定的僵尸
5. 没有剩余目标 → 兜底打最近的（即使已被锁，也比闲着好）
```

用轻量字典 `Dictionary<Zombie, int>` 记录每只僵尸被几架瞄准，Tick 开始前清空。

### 4.9 仇恨系统 🔄（已实现 · 2026-06-01 · v2.0 → v5.0）

> **依赖**: EventBus · **关联**: [[通用NPC寻路系统]] · **设计文档**: `开发计划/仇恨系统.md`

#### 核心理念

像 `DecibelSystem` 一样，提供独立的、可插拔的仇恨/威胁管理体系，解决仇恨逻辑散落各处的问题。

#### 两层架构

```
ThreatSystem (动态仇恨表 · 单例 · 衰减)
    │
    └── 依赖 ──→ FactionSystem (静态阵营关系)
                  ├── IsHostile(a,b) → 🔴
                  ├── IsAlly(a,b)    → 🤝
                  └── IsNeutral(a,b) → 🟡
```

#### FactionSystem（第一层：静态）

- 7 个阵营：Player / Survivor / Zombie / AIBot / Bandit / Military / Neutral
- FactionData SO × 7，配置盟友/敌对列表
- **改为静态类**：`FactionSystem.Initialize(factions[])` → `IsHostile/IsAlly/IsNeutral`

#### ThreatSystem（第二层：动态）

**核心接口**：
- `AddThreat(src, tgt, value, type, pos?)` — 同类型累加 + 上限 500
- `GetTopTarget(src)` — 按 `value × typeWeight` 排序
- `GetLastKnownPosition(src, tgt)` — 目标最后位置
- `DecayAll(dt)` — 按类型衰减（Damage 0.5/s, Visual 2/s, Sound 5/s, AllyDamage 0.3/s）

**ThreatType 权重**：Damage×1.5 > Aggression×1.3 > AllyDamage×1.0 > Visual×0.8 > Territory×0.6 > Sound×0.5

**约束**：单目标上限 500、同时跟踪目标上限 5、友军误伤×0.1

| 组件 | 职责 |
|------|------|
| `FactionComponent` | 挂每个实体，声明阵营；OnEnable 注册到 ThreatSystem + InstanceRegistry |
| `InstanceRegistry` | 静态类，InstanceID→Transform 映射 |

#### 接入方式（EventBus 旁路，不改 IDamageable）

- 伤害：调用方发 `ThreatReportEvent` → ThreatSystem 订阅处理
- 视觉：AIAgent 感知 Tick → `AddThreat(Visual)`
- 声音：AIAgent 基类订阅 NoiseEvent → `AddThreat(Sound)`
- 友军：AIAgent 进 Combat 发 `AllyAlertEvent`（不在 ThreatSystem 内广播）

### 4.10 通用NPC寻路系统（AIAgent） 🔄（已实现 · 2026-06-01）

> **依赖**: ThreatSystem + FactionSystem + DecibelSystem + NavMesh
> **关联**: [[仇恨系统]] · [[僵尸AI设计]]
> **设计文档**: `开发计划/通用NPC寻路系统.md`

#### 核心理念

僵尸、NPC、恶徒、军方都有同一个模式：**听到声音→过去看→看到敌人→追上去→丢失目标→放弃**。抽象为 `AIAgent` 基类，子类只覆写差异化行为。

#### 驱动模式：混合

| 触发类型 | 模式 |
|------|------|
| 威胁出现 → 脱离 Idle | **事件驱动**（立刻反应） |
| 状态内部退出条件 | **每帧 Tick**（丢失视线/到达/超时） |
| 全局中断（受伤/死亡/击晕） | **事件驱动**（任何状态立刻切） |

#### 六状态机

```
Idle          空状态 · Wander/StandBy/Patrol/Follow
   ├─[威胁+看不到]─→ Investigate  走向威胁来源位置，到达后短暂扫描
   ├─[威胁+看到]───→ Combat       追击 + 攻击循环 + 每2s重评估切换目标
   ├─[低血量]─────→ Flee         远离威胁跑至safeDistance
   └─[超出范围]───→ Return       走回activityCenter

全局中断: Stunned(硬直) / Dead(停掉一切)
```

**转移优先级**: Dead > Stunned > Flee > Return > Combat > Investigate > Idle

#### 核心特性

| 特性 | 说明 |
|------|------|
| 三档视觉 | 中心30°/100%识别 → 周边60°/概率 → 余光/仅检测移动 |
| 主动扫描 | 每0.5s OverlapSphere → 逐个CanSee → 喂ThreatSystem |
| 导航恢复 | 卡住检测(2s) → 绕路 → 退后 → 3次失败回Idle |
| Combat切换 | 新目标威胁 > 当前×1.3 才切换（防来回跳动） |
| Idle行为枚举 | Wander(游荡) / StandBy(待命) / Patrol(巡逻) / Follow(跟随) |
| 活动约束 | 所有实体有activityCenter+activityRadius，超出→Return |
| 阵营扩散 | AllyAlertEvent通用化：僵尸扩散→所有阵营都能喊友军 |
| 感知分帧 | 100实体分10组，每帧只Tick 10个 |

###### 4.4.1 ZoneConfig ScriptableObject

新建 `Assets/_Game/Config/ZoneConfig.cs`，一个 SO 定义每个区域类型的 runtime 行为：

```
ZoneConfig (SO)
├─ zoneType: ZoneType (CBD/Residential/Industrial/Green)
├─ displayName: "商业中心"/"住宅区"/"工业区"/"野外"
```

###### 4.4.2 ZoneManager Runtime 单例

```
ZoneManager : MonoBehaviour (单例)

│   └─ 查询该 chunk 中心点落在哪个 district 的 circle 范围内
│       └─ 不在任何 district 内 → Wilderness

│   └─ zoneType → 查 zoneConfigs 字典
└─ 数据来源: WorldGenerator 完成后从 WorldData.districts 拷贝
```

##### 三步打通

```

    ↓ 规则分配 + 手动覆写 → effectiveStation
    ↓ 遍历所有 RecipeData
    ↓ recipe.resultItem → 查图谱节点 → effectiveStation
    ↓ recipe.requiredStation = effectiveStation
    ↓ 保存 .asset
  效果: 改一个物品的工作台覆写→重建图谱→所有产出该物品的配方自动迁移

第二步: 生产设备配方入图谱
  RecipeData 新增字段:
    isIndustrial: bool   // true=设备自动生产, false=手工合成
    productionDeviceName: string  // 归属设备（冲压机/工业炉/车床...）
  
  CreateProductionDevices.cs 中的硬编码配方 →
    转为标准 RecipeData .asset (isIndustrial=true)
    纳入 RecipeCatalog 统一管理
    参与上下游计算（冲压机产通用零件→被高级零件消费）
    
  ProductionMode 自动标记:
    Manual 物品 → 只有手工配方
    Industrial 物品 → 只有工业配方
    Both 物品 → 两种配方都有

第三步: 运行时全部走 ItemGraphManager
  CraftingUI 打开工作台:
    现在: RecipeCatalog.GetByStation(tier)
    改为: ItemGraphManager.GetRecipesForStation(tier)
          → 从图谱查 effectiveStation 匹配的配方
          → 同时支持手工+工业两种模式过滤
  
  ProductionDevice 自运转:
    现在: 硬编码设备配方表
    改为: 查图谱 → GetRecipesForDevice(deviceName)
```

## 🏗️ 开发哲学：依赖优先，逐层添砖

> **一个系统无法独立完成。做到一半发现需要另一个系统做地基，就先去做那个地基。**

### 核心思想

系统之间存在**依赖链**，不可能跳过地基搭二楼：

```
你正在做 僵尸 AI
  └─ 需要 僵尸走路 → 先做 NavMesh 寻路（地基）
  └─ 需要 僵尸打人 → 先做 战斗系统（地基）
  └─ 需要 僵尸掉落 → 先做 世界容器（地基）
```

正确的做法：
1. 先做圆柱体（视觉）
2. 做寻路（地基）→ 僵尸会动了 ✓
3. **暂停**，去做战斗系统（下个地基）
4. 回来做僵尸打人
5. **暂停**，去做世界容器（下个地基）
6. 回来做僵尸掉落

### 进度追踪规则（更新版）

每个模块增加「所需地基」字段，直观看到依赖链：

```
模块名          Phase 1    所需地基（前置系统）
─────────────────────────────────────────────
僵尸 AI         ⬜         ➜ 寻路 + 战斗 + 世界容器
  寻路（NavMesh） ✅       独立（无依赖）
  战斗系统        ⬜        独立（依赖物品数据）
  世界容器        ⬜        ➜ 背包 + 交互
```

### 实际开发流程

```
当前系统 → 检查依赖 → 缺失依赖？
   ├─ 是 → 暂停当前系统，切去搭建依赖地基
   │        地基完成后 → 回到原系统继续
   └─ 否 → 继续开发
```

这样永远不会"卡住"，永远在做一个**可以独立完成**的子任务。

### 4.13 存档系统 📋（设计定稿 · 2026-06-10 修订）

> **定位**: 基础设施，L4 组装层。依赖 EventBus + ChunkManager + 几乎所有游戏系统。
> **核心原则**: 睡觉即存档（类 Project Zomboid），不做 F5 快速存档。存档是风险决策，不是无成本操作。
> **本次修订**: 技术栈从 `JsonUtility + 单JSON文件` 升级为 `SQLite + JSON列`，解决 JsonUtility 不支持 Dictionary/多态 的硬伤，同时获得 Chunk 级别增量读写能力。

---

##### 4.13.1 需求概述

```
玩家的世界投入几小时到几十小时。一旦崩溃/退出/死档，全部消失。
存档系统 = 安全感基础。没有存档 → 没有长期留存。
```

**设计目标**:
1. **完整持久化**: 玩家所有进度不丢失（背包/建造/技能/世界状态）
2. **睡觉触发**: 唯一存档入口 = 床上睡觉（呼应建造描述"用于睡觉恢复体力和保存游戏"）
3. **退出自动存**: 正常退出游戏自动存档，防止忘记
4. **崩溃恢复**: SQLite 事务自带 ACID，写一半崩溃不损坏旧数据
5. **多槽位**: 3 个手动存档槽 + 自动存档 + 退出存档

---

##### 4.13.2 技术选型：为什么 SQLite + JSON 列？

**问题：单 JSON 文件的三个硬伤**

| 硬伤 | 说明 |
|------|------|
| Dict/多态不支持 | JsonUtility 对 `Dictionary<K,V>` 输出空 `{}`，对多态 `ISaveable` 无法还原类型。你的技能表、容器映射、电网连接都是 Dictionary |
| 全量读写 | 每次存档序列化全部 100 Chunk，即使你只改了 1 个。世界从 10×10 扩到 20×20（400 Chunk）后，JSON 可能超过 30MB，睡觉存档时明显卡顿 |
| 原子替换手动实现 | 需要 tmp→backup→rename 三步手动保证崩溃安全，代码脆弱 |

**方案对比**

| | 纯 JSON 文件 | 纯 SQLite 表 | **SQLite + JSON 列 ✅** |
|---|---|---|---|
| 写入方式 | 全量 7MB | 行级增量 | 列级增量 |
| 读取方式 | 全量加载 | 按需查询 | 按需查询 |
| 数据格式约束 | 无 | Schema 严格，改表结构痛苦 | **JSON 内自由，加字段不迁移** |
| 可调试性 | 文本编辑器 | 需 SQLite 浏览器 | SELECT 出来就是 JSON |
| 崩溃安全 | 手动实现 | 事务自带 | 事务自带 |
| Chunk 分片 | 自己解析 | `WHERE chunk_id=5` | `WHERE chunk_id=5` |
| 依赖 | 零（内置） | sqlite-net | sqlite-net + Newtonsoft.Json |
| 复杂度 | 低 | 高 | **中** |

**结论**: SQLite 管理"怎么存、怎么查"（增量、分片、事务），JSON 管理"存什么格式"（灵活、可读、无 Schema 约束），互补长短。

**依赖包**:
```

sqlite-net                         # SQLite 轻量 ORM（Frank Krueger / praeclarum）
```

**为什么 backup 是文件级复制而不是 SQLite 内置？**
- SQLite `.backup` API 可以办到，但实现更复杂
- 存档时 `File.Copy(save.db, save.db.backup)` + 再写 `save.db` 最简单
- SQLite 事务保证 `save.db` 要么写完整要么无效，`save.db.backup` 始终是上一版成功存档
- 两版都坏的概率极低（磁盘满/断电同时发生）

##### 4.13.5 数据库 Schema

```sql
-- ============================================
-- 存档元信息
-- ============================================
CREATE TABLE save_info (
    key   TEXT PRIMARY KEY,
    value TEXT
);
-- 存储: version, created_at, play_time, scene_name, world_seed

-- ============================================
-- 玩家数据（单行）
-- ============================================
CREATE TABLE player (
    id      INTEGER PRIMARY KEY DEFAULT 1,
    data    TEXT NOT NULL,   -- JSON: PlayerSaveData
    saved_at TEXT
);

-- ============================================
-- Chunk 级数据（4 张表，按 chunk_id 分片）
-- ============================================
CREATE TABLE chunk_structures (
    chunk_id    INTEGER PRIMARY KEY,
    buildings   TEXT,        -- JSON: PlacedStructure[]
    power_grid  TEXT,        -- JSON: PowerNetworkData
    devices     TEXT,        -- JSON: ProductionDevice[]
    updated_at  TEXT
);

CREATE TABLE chunk_containers (
    chunk_id     INTEGER PRIMARY KEY,
    records      TEXT,       -- JSON: ContainerRecord[]
    updated_at   TEXT
);

CREATE TABLE chunk_world_items (
    chunk_id    INTEGER PRIMARY KEY,
    items       TEXT,        -- JSON: WorldItemData[]
    updated_at  TEXT
);

CREATE TABLE chunk_meta (
    chunk_id          INTEGER PRIMARY KEY,
    state             INTEGER,   -- 0=Inactive, 1=Preloaded, 2=Loaded
    refresh_cooldown  REAL,      -- 剩余冷却天数
    opened_game_day   REAL,      -- 最后搜索的游戏天数
    updated_at        TEXT
);

-- ============================================
-- 独立实体数据
-- ============================================
CREATE TABLE vehicles (
    vehicle_id  INTEGER PRIMARY KEY,
    chunk_id    INTEGER,      -- 所属 Chunk（Chunk 卸载时按此查询）
    data        TEXT,         -- JSON: VehicleSaveData
    updated_at  TEXT
);
CREATE INDEX idx_vehicles_chunk ON vehicles(chunk_id);

CREATE TABLE ai_bots (
    bot_id      INTEGER PRIMARY KEY,
    chunk_id    INTEGER,
    data        TEXT,         -- JSON: AIBotSaveData
    updated_at  TEXT
);
CREATE INDEX idx_aibots_chunk ON ai_bots(chunk_id);

-- ============================================
-- 系统状态（Key-Value）
-- ============================================
CREATE TABLE system_state (
    key   TEXT PRIMARY KEY,
    value TEXT       -- JSON
);
-- 存储 game_time / weather / threat / 等各系统状态
```

**Schema 设计原则**:
1. 每张 Chunk 表只存一个 Chunk 的一行，无关联查询，读写都是单行操作
2. JSON 列内字段自由增删，不需要 ALTER TABLE
3. 独立实体（车、机器人）有独立的表和自己的 chunk_id，Chunk 卸载时按索引快速定位
4. `system_state` 用 key-value 存所有不按 Chunk 分片的状态

**接口设计**:

```csharp
// 任何需要持久化的系统实现此接口
public interface ISaveable
{
    string SaveId { get; }
    void OnSave(SaveContext ctx);    // 系统自行决定写哪些数据
    void OnLoad(SaveContext ctx);    // 系统自行决定读哪些数据
}

// SaveContext 封装所有数据库读写，各系统通过它操作
public class SaveContext
{
    public int CurrentSlot { get; }
    
    // 玩家
    public void WritePlayer<T>(T data);
    public T ReadPlayer<T>();
    
    // Chunk 级（按 chunk_id 分片读写）
    public void WriteChunkStructures(int chunkId, ChunkStructureData data);
    public ChunkStructureData ReadChunkStructures(int chunkId);
    public void WriteChunkContainers(int chunkId, ContainerData data);
    public ContainerData ReadChunkContainers(int chunkId);
    public void WriteChunkWorldItems(int chunkId, WorldItemData data);
    public WorldItemData ReadChunkWorldItems(int chunkId);
    public void WriteChunkMeta(int chunkId, ChunkMetaData data);
    public ChunkMetaData ReadChunkMeta(int chunkId);
    
    // 独立实体
    public void WriteVehicle(int vehicleId, int chunkId, VehicleSaveData data);
    public VehicleSaveData ReadVehicle(int vehicleId);
    public List<VehicleSaveData> ReadAllVehicles();
    public void WriteAIBot(int botId, int chunkId, AIBotSaveData data);
    public AIBotSaveData ReadAIBot(int botId);
    public List<AIBotSaveData> ReadAllAIBots();
    
    // 系统状态
    public void WriteSystemState<T>(string key, T data);
    public T ReadSystemState<T>(string key);
    
    // 存档元信息
    public void WriteMeta(string key, string value);
    public string ReadMeta(string key);
}
```

##### 4.13.7 双轨制存档流程

```
┌──────────────────────────────────────────────────────────┐
│                    存档系统双轨制                          │
│                                                          │
│  轨道 ① 事件驱动增量写（运行时，每次玩家动作后立即）          │
│  ────────────────────────────────────────                 │
│  StructurePlacedEvent → SaveManager 收到 →                 │
│    ctx.WriteChunkStructures(chunkId, 该Chunk当前数据)      │
│    └── 只写 1 行，~1ms                                    │
│                                                          │
│  ContainerOpenedEvent → SaveManager 收到 →                 │
│    ctx.WriteChunkContainers(chunkId, 该Chunk当前数据)      │
│    └── 只写 1 行，~1ms                                    │
│                                                          │
│  轨道 ② 定时/触发全量提交（睡觉/退出/定时器）                  │
│  ────────────────────────────────────────                 │
│  SaveManager.SaveAll():                                   │
│    遍历 ISaveable → OnSave(ctx)                           │
│    └── 大部分系统的 OnSave 查不到变化就跳过（无操作）         │
│    └── 只有变更过的数据才重新序列化写入                      │
│    写入 save_info（版本/时间戳等元信息）                     │
│    File.Copy(save.db, save.db.backup)                     │
│    写入 meta.json                                         │
│    Publish(SaveCompletedEvent)                            │
│                                                          │
│  关键认知：轨道①已经写过了，轨道②只提交脏数据                  │
│  睡觉时真正的写入量 = 轨道①没覆盖的（玩家HP/时间/天气等）      │
│  ≈ 几 KB，远小于全量 7MB，黑屏过渡完全遮得住                 │
└──────────────────────────────────────────────────────────┘
```

**加载流程**:

```
1. 打开 save.db
2. 读取 save_info 校验版本兼容
3. 初始化世界: WorldGenerator.Run(seed) 重建地形
4. 遍历 ISaveable → OnLoad(ctx):
   - PlayerCharacter: ReadPlayer() 恢复位置/装备/背包
   - ChunkManager: ReadChunkMeta() 知道哪些 Chunk 被修改过
   - ChunkManager 加载 Chunk 时:
     if (ReadChunkStructures(chunkId) != null) 恢复建造物
     if (ReadChunkContainers(chunkId) != null) 恢复容器
     if (ReadChunkWorldItems(chunkId) != null) 恢复地面物品
     else 用种子+LootTable 重新生成（玩家没碰过的 Chunk）
5. 重建非持久化状态（NavMesh/电网拓扑/视觉）
6. Publish(LoadCompletedEvent)

关键: 读档后世界 = 种子生成的基础世界 + 玩家的 Delta 修改
      没被玩家碰过的 Chunk → 按种子重生成（保证一致）
      被玩家碰过的 Chunk → 从存档恢复（保留玩家改动）
```

##### 4.13.12 边界条件与兼容性

| 场景 | 处理 |
|------|------|
| 存档版本升级 | `save_info.version` 递增。打开旧版 db 时执行版本迁移函数。JSON 列内字段宽容读取（多字段忽略，缺字段用默认值） |
| 物品 ItemData 改名/删除 | 按 itemName 匹配，找不到 → 丢弃该物品 + 日志警告 |
| BuildableData 删除 | 读档时找不到的建造物 → 返还材料到玩家背包 |
| save.db 损坏 | 尝试从 `save.db.backup` 恢复。两者都坏 → 提示"存档损坏，是否删除并新建？" |
| 存档空间不足 | SQLite 写入前不检查空间。写入失败 catch SQLiteException → 回滚 + 提示 |
| 读档时场景未加载 | 先加载场景 → WorldGenerator 重建世界 → 再逐 Chunk 应用存档 Delta |
| Chunk 从未被玩家碰过 | `chunk_structures` / `chunk_containers` 表无该行 → 跳过，用种子重生成 |
| WAL 模式并发 | SQLite 开启 WAL（Write-Ahead Logging）模式，允许多个读取者同时进行。增量写入+SaveAll 不冲突 |

### 4.14 对象池系统 📋（设计定稿 · 待实施）

> **定位**: 性能基础设施，L4 组装层。依赖: 无（独立工具类）。
> **核心原则**: 高频创建/销毁的对象必须走池。不做过度设计，只池真正需要池的。

---

##### 4.14.1 为什么需要对象池

```
当前问题:
- 僵尸生成 = Instantiate(僵尸prefab)    ← GC 分配
- 僵尸死亡 = Destroy(gameObject)        ← GC 释放
- 子弹发射 = Instantiate + Destroy      ← 高频 GC
- 地面物品 = Instantiate + Destroy      ← 大量 GC
- 粒子特效 = Instantiate + Destroy      ← 极高频 GC

结果: 帧率周期性卡顿（GC.Collect），大量对象 → 内存碎片。
```

**池化收益估算**:

| 对象类型 | 频率 | 生命周期 | 池化收益 |
|------|:--:|:--:|:--:|
| 僵尸 | 中 (10~50/min) | 长 (分钟级) | ⭐⭐⭐ 高 — 减少 GC 尖刺 |
| 子弹/投射物 | 高 (100~500/min) | 短 (秒级) | ⭐⭐⭐⭐⭐ 极高 — 消除每帧分配 |
| 地面物品 | 中 (掉落频率) | 长 (直到拾取) | ⭐⭐ 中 |
| 粒子特效 | 极高 (击中/血/爆炸) | 极短 (<1s) | ⭐⭐⭐⭐⭐ 极高 |
| 音效发射器 | 中 | 短 (秒级) | ⭐⭐ 中 |
| 伤害数字 UI | 高 | 极短 (<1s) | ⭐⭐⭐ 高 |

##### 4.14.2 架构设计

**核心组件（2 新建文件）**:


**ObjectPool\<T\> 设计**:

```csharp
/// <summary>
/// 泛型对象池。T 必须是 Component。
/// 线程不安全，仅主线程使用。
/// </summary>
public class ObjectPool<T> where T : Component
{
    // === 配置 ===
    public T Prefab { get; }           // 池化的预制体
    public int MaxSize { get; }        // 池子上限（超出后直接 Destroy）
    public float ShrinkInterval { get; } // 缩容检查间隔（秒）
    
    // === 运行时状态 ===
    public int ActiveCount { get; }    // 当前活跃对象数
    public int InactiveCount { get; }  // 池中待命对象数
    public int PeakActive { get; }     // 历史最高活跃数
    public int TotalSpawned { get; }   // 累计 Get 次数
    
    // === 核心 API ===
    T Get(Vector3 position, Quaternion rotation);    // 从池取/实例化
    T Get(Vector3 position);                         // 简化版（无旋转）
    void Release(T obj);                             // 归还池中
    void Prewarm(int count);                         // 预热
    void Shrink(int targetCount);                    // 缩容到目标数
    void Clear();                                    // 清空池（场景切换用）
}
```

**实现要点**:

```
Get:
  1. 如果池中有闲置对象 → 取最后一个（Stack.Pop，O(1)），激活 + 设位置
  2. 如果池空 + ActiveCount < MaxSize → Instantiate 新对象
  3. 如果池空 + ActiveCount >= MaxSize → 找最早活跃对象强制回收再分配（可选策略）
  4. 设置 GameObject.SetActive(true) + 重置 Transform

Release:
  1. 对象入栈（Stack.Push）
  2. 设置 GameObject.SetActive(false)
  3. 可选：将对象挂到池子根节点下（Scene 整洁）
  4. 如果 InactiveCount > 缩容阈值 → 标记缩容

缩容:
  1. 定时器触发（ShrinkInterval）
  2. 销毁超出 targetCount 的多余对象（Destroy）
  3. 只在池闲置对象数 > MaxSize 的 50% 时触发（避免频繁缩容）
```

**PoolManager 设计**:

```csharp
/// <summary>
/// 全局池管理器。单例，负责所有对象池的注册/统计/场景切换清理。
/// </summary>
public class PoolManager : MonoBehaviour
{
    // === 全局 API ===
    public static PoolManager Instance { get; }

    // 快捷方法（不用每次传 prefab）

    // 预热
    public void Prewarm<T>(T prefab, int count) where T : Component;
    
    // 场景清理

    // 调试

}
```

##### 4.14.6 注意事项

| 项 | 说明 |
|------|------|
| **归还时重置** | Release 时必须重置对象状态（HP/动画/速度/颜色），否则下次 Get 拿到"脏"对象 |
| **OnEnable/OnDisable** | 依赖 Awake/Start 做初始化的组件，改为依赖 OnEnable（每次从池取出触发） |
| **池上限** | 超过 MaxSize 且池中无闲置 → 扩容（不卡），而非报错。MaxSize 是"缩容目标"不是"硬上限" |
| **场景切换** | PoolManager 必须是 DontDestroyOnLoad，场景切换时 ClearAll() |
| **预制体匹配** | 每个池绑定特定预制体，不同僵尸类型 = 不同池 |
| **不池化 UI 面板** | 只池化频繁创建/销毁的小 UI 元素（伤害数字），不池化大面板（背包/合成/HUD） |

### 🖥️ UI系统UGUI重构计划

> **日期**: 2026-06-10 · **状态**: 📋 设计定稿 · **依赖**: 无（纯UI层改造，不碰系统逻辑）
> **目标**: 消除全部运行时 IMGUI（`OnGUI()`），统一到 UGUI Canvas 体系

#### 现状

| 系统 | 文件数 | 用途 |
|------|--------|------|
| UGUI (`UnityEngine.UI`) | 8 | 背包、HUD条、拖拽、交互提示、容器窗口 |
| IMGUI (`OnGUI()`) | 14 | 状态HUD、合成、建造、电力、AI机器人、调试 |
| UI Toolkit | 0 | — |

两套系统通过每个面板里的 `CloseOtherUIs()` 手动互斥，无统一管理层。

#### 梯度一：简单迁移（~1.5 天）

只换渲染层，不改逻辑。`GUI.Label`→`Canvas Text`，`GUI.Button`→`UGUI Button`。

| 文件 | 方案 |
|------|------|
| `DecibelHUD.cs` | Canvas Text 噪声等级 |
| `WeatherHUD.cs` | Canvas Text 天气 |
| `TopLeftHUD.cs` | Canvas Text 左上状态（时间/车速/武器） |
| `DevTools.cs` | Canvas + VerticalLayout 按钮列表 |
| `DebugPanel.cs` | Canvas Text 面板 |
| `ZombieSpawnDebugWindow.cs` | Canvas Panel + Slider |
| `PowerSourceUI.cs` | Canvas Image + Text（电源状态/燃料条/按钮） |

#### 梯度二：中等迁移（~3 天）

需要 ScrollRect + GridLayout 替换滚动列表，核心逻辑不动。

| 文件 | 方案 |
|------|------|
| `CraftingUI.cs` | ScrollRect + GridLayout 配方列表 + 材料预览（参考现有 InventoryUI 动态格子模式） |
| `ProductionDeviceUI.cs` | 同上 + 设备链接按钮 |
| `ChemicalResearchUI.cs` | ScrollRect 科技树面板 |
| `BuildMenuUI.cs` | GridLayout + 按钮预制体（标签栏+物品网格+材料面板） |
| `TerminalUI.cs` | ScrollRect + Text（电网数据） |

#### 梯度三：复杂迁移（~3 天）

`AIBotUI` 的 6 个 `GUI.Window` 重构为 UGUI Panel 栈，物品渲染复用 InventoryUI 的格子生成逻辑。

| 文件 | 方案 |
|------|------|
| `AIBotUI.cs` | 主面板 + 3 设置 Panel + 武器管理 Panel + 背包 Panel（`RectTransform` 拖动替代 `GUI.DragWindow`） |
| `AIBotInventoryUI.cs` | 直接用现有 InventoryUI 网格模式 |

#### 不会丢失的功能

- **滚动列表** → `ScrollRect`（比 `BeginScrollView` 流畅，无 GC 分配）
- **拖拽窗口** → `IPointerDownHandler` + `IDragHandler`（已有 `ItemDragHandler` 先例）
- **动态物品格子** → `InventoryUI.cs` 已验证（GameObject + Image + Text）
- **实时更新** → Canvas 脏标记驱动，不再每帧 `OnGUI()` 分配 GC
- **输入框/按钮/滑块** → UGUI 原生组件

#### 实施策略

- 每个梯度独立可验证，不跨梯度依赖
- 梯度一完成 → 删掉 7 个 `OnGUI()`，HUD 全 UGUI
- 梯度二完成 → 游戏不再有运行时 IMGUI
- 梯度三完成 → 全 UI 体系统一
- 保留 `DevTools.cs` 为 IMGUI 可接受（仅开发用，正式版关闭）
