## 🔄 阶段 2：世界骨架（代码 + 数据）

> 核心原则：**按依赖层级开发，不跳级**。
> L1 → L2 → L3 → L4，每一层为上一层提供材料。

### 层级总图

```
L1 地基 ─── 任何系统都依赖它
  ├─ ItemData (SO) ✅     物品是一切的基础
  ├─ EventBus ✅          系统间通信
  ├─ 移动/摄像机 ✅       玩家操作基础
  └─ 碰撞/物理 ✅         世界交互基础

L2 框架 ─── L3/L4 都要用它
  ├─ 背包系统 ✅          所有物品存放的地方
  ├─ 装备系统 ✅          Dictionary数据驱动，容器尺寸、属性加成
  ├─ 交互系统 ✅          按 E 与世界互动
  ├─ 时间系统 ✅          日夜循环、Tick 驱动
  ├─ 人物系统 ✅          性别/职业/属性/成长
  └─ 物品使用系统 ✅      ItemUsageSystem，解耦UI与系统层

L3 机制 ─── 具体的玩法系统（由 L2 驱动）
  ├─ 世界容器 ✅          柜子/冰箱/尸体（地表物品✅, Phase2完成）
  ├─ 生存数值 ✅          Phase1核心完成
  ├─ 战斗系统 ✅          Phase1~6全完成（武器数据+装备+模型+瞄准+射击+伤害）
  ├─ 建造系统 ✅          Phase1完成（Ghost预览+放置+拆除+BuildMenuUI+7默认资产）
  ├─ 交通工具 ✅          Phase1完成（驾驶+上车/下车+摄像机跟随+油量+SHIFT加速+防侧翻）
  └─ 天气系统 ✅          Phase1完成（马尔可夫链+场景光照+HUD+DevTools）

L4 组装 ─── 需要多个 L3 配合
  ├─ ChunkManager ✅     Phase1完成（三级区块+预热队列+RefreshHub+性能分档）
  ├─ ContainerRegistry ✅ Phase1完成（懒加载+两态追踪+区块刷新判定）
  ├─ 僵尸 AI ✅           Phase1+2完成（FSM+NavMesh+集中感知+ZombieData+刷怪）
  ├─ 配方/制作 ✅          Phase1~4完成（159配方+19生产设备+流水线+电力集成）
  ├─ 电力/水源 ✅          Phase1完成（8电力设备+电网拓扑+双模驱动+燃料体系）
  └─ 区域/建筑 ⬜         分区/Loot/建筑模板（依赖 ChunkManager + ContainerRegistry）
```

#### 4.3 电力系统 ✅（Phase1 完成）

##### 设计哲学

**不铺电路，不模拟电压电流。只做功率匹配。**

```
总输出(W) ≥ 总负载(W) = 全部运转
总输出(W) < 总负载(W) = 全网红灯，玩家自行解决
```

##### 电网拓扑：两级连接

```
发电端 ──[电缆]──→ 用电终端 ──[圆形范围]──→ 设备自动供电
                     │
                     └──[电缆]──→ 另一个终端 ──[圆形范围]──→ 更多设备
```

- **发电端 → 终端**：需要拉一条电缆（有建造感，但不繁琐）
- **终端 → 终端**：可以串联，全电网共享一个总功率池
- **终端 → 设备**：圆形范围内自动供电，无需逐设备接线
- **电网定义**：所有用电缆连在一起的终端 + 它们接入的发电端 = 一个电网
- **功率更新**：任何终端接入新发电端 → 整个电网的功率池自动更新

##### 组件架构（3 新建文件）

```csharp
// Systems/Power/PowerSource.cs — 发电端
public class PowerSource : MonoBehaviour
{
    public float maxOutput;          // 最大输出(W)
    public PowerSourceType sourceType; // Water/Wind/Combustion/Solar/Human
    public float fuelConsumptionRate; // 燃料消耗率（燃烧型）
    public bool isActive;
}

// Systems/Power/PowerTerminal.cs — 用电终端（原 PowerConsumer 改为终端概念）
public class PowerTerminal : MonoBehaviour, IInteractable
{
    public float supplyRadius = 15f;  // 供电半径
    public List<PowerSource> connectedSources;
    public List<PowerTerminal> connectedTerminals;
    public float totalGridPower;      // 所属电网总功率
    public float currentLoad;         // 本终端范围内设备总负载
    
    void RecalculateGrid();           // 终端接入/断开时重新计算全网功率
}

// Systems/Power/PowerConsumer.cs — 设备端
public class PowerConsumer : MonoBehaviour
{
    public float requiredPower;       // 需功率(W)
    public bool isPowered;            // 是否在供电范围内+电网功率充足
    public bool allowCoal;            // 是否允许烧煤兜底
}
```

##### 核心问题

世界生成管线算出了 districts/blocks/cityBuildings，但这份数据只存在 `WorldData` 里，**runtime 系统（ChunkManager、ZombieSpawner、ContainerRefresh）完全不消费它**。等于说区域数据"算完就丢了"。

##### 目标

打通 **WorldData → Runtime 系统** 的传递链，让容器刷新和僵尸生成能按真实区域类型读取不同配置。

---

##### 子任务拆解

###### 4.4.1 ZoneConfig ScriptableObject

新建 `Assets/_Game/Config/ZoneConfig.cs`，一个 SO 定义每个区域类型的 runtime 行为：

```
ZoneConfig (SO)
├─ zoneType: ZoneType (CBD/Residential/Industrial/Green)
├─ displayName: "商业中心"/"住宅区"/"工业区"/"野外"
├─ spawnProfile: ZoneSpawnProfile 引用 (僵尸数量/类型权重)
├─ defaultLootTable: LootTable 引用 (容器默认掉落)
└─ lootMultiplier: float (1.0=标准, 商业区可设1.5, 野外0.5)
```

Runtime 时只有一个 `ZoneConfig[]` 数组（或 Dictionary<ZoneType, ZoneConfig>），挂在 ZoneManager 上。

**荒野/野外**：当前 ZoneType 枚举没有 Wilderness，但 Green（绿地/公园）不等于野外。新增 `Wilderness` 枚举值到 `ZoneType`，代表城市外的野外区域。Chunk 不在任何 district 内时 = Wilderness。

###### 4.4.2 ZoneManager Runtime 单例

新建 `Assets/_Game/Systems/WorldGen/ZoneManager.cs`：

```
ZoneManager : MonoBehaviour (单例)
├─ zoneConfigs: ZoneConfig[] (Inspector 拖入)
├─ GetZoneType(chunkId): ZoneType
│   └─ 查询该 chunk 中心点落在哪个 district 的 circle 范围内
│       └─ 不在任何 district 内 → Wilderness
├─ GetZoneConfig(chunkId): ZoneConfig
│   └─ zoneType → 查 zoneConfigs 字典
└─ 数据来源: WorldGenerator 完成后从 WorldData.districts 拷贝
```

**查区域逻辑**：chunk 中心世界坐标 → 遍历所有 CityDistrict → 判断 `Vector2.Distance(chunkCenter, district.center) <= district.radius` → 命中则返回 district.zoneType。多个 district 重叠时取距离最近的。

###### 4.4.3 RuntimeChunk 加 zoneType

`RuntimeChunk.cs` 新增字段：

```csharp
public ZoneType zoneType;  // 由 ZoneManager 在 chunk 首次加载时写入
```

ChunkManager 在 chunk 从 Preload 进入 Loaded 时，调用 `ZoneManager.Instance.AssignZone(chunk)`，写入 `chunk.zoneType`。

###### 4.4.4 ZombieSpawner 接入真实区域

修改 `ZombieSpawner.GetProfile(int chunkId)`：

```
现状: hash 伪随机 → zoneProfiles[index]
改为: ZoneManager.Instance.GetZoneConfig(chunkId)?.spawnProfile
```

- **CBD**：高密度（initialMin=4, maxPerChunk=12），快速僵尸权重高
- **Residential**：中密度（initialMin=2, maxPerChunk=8），普通僵尸为主

### 4.9 物品追踪系统 → 配方/工作台全面接入 ✅（已完成 · 2026-05-29）

##### 三步打通

```
第一步: 图谱 → 配方同步（构建时自动）
  构建图谱
    ↓ 规则分配 + 手动覆写 → effectiveStation
    ↓ 遍历所有 RecipeData
    ↓ recipe.resultItem → 查图谱节点 → effectiveStation
    ↓ recipe.requiredStation = effectiveStation
    ↓ 保存 .asset
  效果: 改一个物品的工作台覆写→重建图谱→所有产出该物品的配方自动迁移

第二步: 生产设备配方入图谱
  RecipeData 新增字段:
    isIndustrial: bool   // true=设备自动生产, false=手工合成
    productionDeviceName: string  // 归属设备（冲压机/工业炉/车床...）
  
  CreateProductionDevices.cs 中的硬编码配方 →
    转为标准 RecipeData .asset (isIndustrial=true)
    纳入 RecipeCatalog 统一管理

  ProductionMode 自动标记:
    Manual 物品 → 只有手工配方
    Industrial 物品 → 只有工业配方
    Both 物品 → 两种配方都有

第三步: 运行时全部走 ItemGraphManager
  CraftingUI 打开工作台:
    现在: RecipeCatalog.GetByStation(tier)
    改为: ItemGraphManager.GetRecipesForStation(tier)
          → 从图谱查 effectiveStation 匹配的配方
          → 同时支持手工+工业两种模式过滤

    现在: 硬编码设备配方表

```

##### 四、化学台"配方解锁"机制（新系统）

化学台不再是生产设备，而是**研究解锁**：

```
化学台界面:
┌──────────────────────────────────────────┐
│  🔬 化学工艺研究                          │
│                                          │
│  已解锁: 发酵工艺 ✓  蒸馏工艺 ✓          │
│  研究中: 电解工艺 [硫酸+化学试剂×3]      │
│  待研究: 制药工艺  火药工艺  离心工艺    │
│                                          │
│  研究消耗材料，解锁后对应工业设备可生产   │
└──────────────────────────────────────────┘
```

解锁一个工艺后，对应的工业设备才能运转。一次解锁，永久生效。

## 🏗️ 开发哲学：依赖优先，逐层添砖

> **一个系统无法独立完成。做到一半发现需要另一个系统做地基，就先去做那个地基。**

### 核心思想

系统之间存在**依赖链**，不可能跳过地基搭二楼：

```
你正在做 僵尸 AI
  └─ 需要 僵尸走路 → 先做 NavMesh 寻路（地基）
  └─ 需要 僵尸打人 → 先做 战斗系统（地基）
  └─ 需要 僵尸掉落 → 先做 世界容器（地基）
```

正确的做法：
1. 先做圆柱体（视觉）
2. 做寻路（地基）→ 僵尸会动了 ✓
3. **暂停**，去做战斗系统（下个地基）
4. 回来做僵尸打人
5. **暂停**，去做世界容器（下个地基）
6. 回来做僵尸掉落

### 进度追踪规则（更新版）

每个模块增加「所需地基」字段，直观看到依赖链：

```
模块名          Phase 1    所需地基（前置系统）
─────────────────────────────────────────────
僵尸 AI         ⬜         ➜ 寻路 + 战斗 + 世界容器
  寻路（NavMesh） ✅       独立（无依赖）
  战斗系统        ⬜        独立（依赖物品数据）
  世界容器        ⬜        ➜ 背包 + 交互
```

### 实际开发流程

```
当前系统 → 检查依赖 → 缺失依赖？
   ├─ 是 → 暂停当前系统，切去搭建依赖地基
   │        地基完成后 → 回到原系统继续
   └─ 否 → 继续开发
```

这样永远不会"卡住"，永远在做一个**可以独立完成**的子任务。
