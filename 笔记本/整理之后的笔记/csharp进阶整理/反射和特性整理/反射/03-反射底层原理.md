# 反射（Reflection）—— Step 3：理解为什么（底层原理）

---

## 3.1 普通调用 vs 反射调用

**普通方式调用流程：**
```
编译时：编译器"看到" Monster 类 → 知道 Attack 方法的地址
       → 直接编译成：jump 0x0040A230（跳转指令）

运行时：CPU 直接跳转到那个地址执行 → 完成
```

**反射方式调用流程：**
```
编译时：编译器不知道运行时想调用哪个方法
       → 生成"运行时查找"的指令

运行时（5步走）：
  ① 从程序集的元数据区读取 Monster 类的元数据
  ② 在元数据里按名字查找 Attack 方法的位置
  ③ 验证参数类型是否匹配（int）
  ④ 找到方法地址
  ⑤ 跳转执行
```

## 3.2 元数据表详解

### 程序集文件内部结构

```
┌────────────────────────────────────┐
│           PE 头                     │  ← Windows 可执行文件格式头
├────────────────────────────────────┤
│           CLR 头                    │  ← .NET 特有信息
├────────────────────────────────────┤
│     IL 代码区                       │  ← 实际执行的指令
├────────────────────────────────────┤
│     元数据区（Reflection 读的就是这里）│
│  ├── #~ 流（压缩表流）              │
│  │     各种元数据表                  │
│  ├── #Strings 堆（存 "Attack" 等字符串）│
│  ├── #Blob 堆（存签名）              │
│  ├── #GUID 堆                       │
│  └── #US 堆（用户字符串）            │
└────────────────────────────────────┘
```

### 元数据里的表

| 表名 | 存什么 | 反射对应 |
|---|---|---|
| `TypeDef` | 类的定义（名字、基类） | `Type` |
| `MethodDef` | 方法定义（名字、签名、IL 偏移） | `MethodInfo` |
| `FieldDef` | 字段定义（名字、类型、偏移） | `FieldInfo` |
| `Property` | 属性定义 | `PropertyInfo` |
| `ParamDef` | 参数定义 | `ParameterInfo` |

### GetMethod 底层查找过程

```
GetMethod("Attack") 查找过程：

① 读取 TypeDef 表 → 找到 Monster 类，得到 MethodDef 表索引范围

② 遍历 MethodDef 表（每一行格式如下）
   ┌──────────────────────────────────┐
   │ 名字（字符串堆偏移）              │
   │ 标志（public/private/static...）  │
   │ 签名（Blob 堆偏移）               │
   │ IL 代码偏移                       │
   └──────────────────────────────────┘
   → 按名字匹配 "Attack"
   → 匹配成功，读取 IL 偏移地址

③ 返回 MethodInfo 对象
   { Name = "Attack", ILAddress = 0x0040A230 }

④ Invoke 时，CLR 跳转到 ILAddress 执行
```

## 3.3 装箱拆箱的内存布局

### 值类型在内存里

```csharp
int hp = 100;
```

```
栈内存：
┌────────────────┐
│     hp          │  ← 直接存着值 100
│   (4字节)       │
└────────────────┘
```

### 装箱时发生了什么

```csharp
int hp = 100;
object box = hp; // 装箱
```

```
装箱前：栈上
┌────────────────┐
│     hp = 100    │
└────────────────┘

装箱时（堆上分配新对象）：
┌─────────────────────────────────┐
│  对象头（8字节）: 类型指针 + 同步块 │
├─────────────────────────────────┤
│        100（存的就是值）          │
└─────────────────────────────────┘
    ↑ object box 指向这里

装箱做了三件事：
① 在堆上分配新对象
② 把值复制进去
③ 返回对象引用
```

### 拆箱时发生了什么

```csharp
int hp2 = (int)box; // 拆箱
```

```
拆箱过程：
① 检查堆上那个对象确实是 int 类型（类型检查）
② 把堆上的值（100）复制回栈上的 hp2
```

### 数组的装箱问题（重要！）

```csharp
int[] arr = new int[] { 1, 2, 3 };     // 无装箱，int 直接存
object[] arr2 = new object[] { 1, 2, 3 }; // 每个 int 都单独装箱！
```

```
arr2 的内存：
堆：
├─ 数组头
├─ object(1) → int对象(1)  ← 装箱了！
├─ object(2) → int对象(2)  ← 装箱了！
└─ object(3) → int对象(3)  ← 装箱了！
```

## 3.4 反射性能详解

### 性能开销来源

```
性能开销分三层：

① 查找开销（GetMethod / GetField）
   ├── 遍历元数据表（O(n)）
   ├── 字符串比较匹配
   └── 一次反射可能查找几十上百次

② 调用开销（Invoke）
   ├── 参数装箱（int → object）
   ├── 返回值拆箱（object → int）
   ├── 类型安全验证（每次都要）
   └── 无法 JIT 内联优化

③ GC 压力
   ├── 装箱产生堆对象 → 触发 GC
   └── object[] 数组也是堆对象
```

### 性能对比

```
调用 10000 次方法：

普通调用：        1ms
反射调用（无缓存）：500ms  ← 500倍差距！
反射调用（有缓存）：10ms  ← 改善很大
Delegate 委托：    1ms    ← 和普通调用一样快
```

### 什么时候反射慢？

| 场景 | 问题 |
|---|---|
| 每帧 Update 里用反射 | 60fps × 反射开销 = 卡顿 |
| 循环里反复 GetMethod | 查找次数爆炸 |
| 热路径（性能关键路径） | 影响帧率 |

### 优化方案一：缓存 Info 对象

```csharp
// ❌ 每次都查找，慢
void Update()
{
    MethodInfo m = t.GetMethod("Attack"); // 每帧都查！
    m.Invoke(obj, new object[]{ 50 });
}

// ✅ 只查找一次，后续复用，快
MethodInfo attackMethod;

void Start()
{
    attackMethod = t.GetMethod("Attack"); // 只查一次
}

void Update()
{
    attackMethod.Invoke(obj, new object[]{ 50 }); // 后续直接用
}
```

### 优化方案二：转成委托（最快）

```csharp
void Start()
{
    // 只反射一次，之后就变成普通委托调用
    attackDelegate = (Action<int>)Delegate.CreateDelegate(
        typeof(Action<int>),    // 参数1：委托类型（创建什么类型的委托）
        monster,                // 参数2：实例对象（绑定到哪个实例）
        t.GetMethod("Attack")  // 参数3：方法（绑定哪个方法）
    );
}

Action<int> attackDelegate; // 缓存起来

void Update()
{
    attackDelegate(50); // 和普通调用一样快！
}
```

### CreateDelegate 语法拆解

```
                                    三个参数
                                        ↓
攻击委托  =  (Action<int>)Delegate.CreateDelegate(
                                    typeof(Action<int>),   ← 参数1：创建什么类型的委托
                                    monster,                ← 参数2：绑定到哪个实例
                                    t.GetMethod("Attack")  ← 参数3：绑定哪个方法
                                  );

↑
强转成 Action<int>
因为 CreateDelegate 返回的是 Delegate 基类
需要强转成具体委托类型才能直接调用
```

| 参数 | 作用 | 说明 |
|---|---|---|
| `typeof(Action<int>)` | 委托类型 | 告诉它要创建什么类型的委托 |
| `monster` | 实例对象 | 绑定到哪个实例（静态方法传 null） |
| `t.GetMethod("Attack")` | 方法 | 绑定哪个方法 |

### 性能总结

```
性能排名（快→慢）：
1. 普通调用 m.Attack(50)           ← 最快
2. 委托调用 delegate()             ← 一样快
3. 反射+缓存（一次查找+多次调用）  ← 可接受
4. 反射无缓存（每次查找+每次调用） ← 慢

优化原则：
├── 只查一次，后续复用（缓存 MethodInfo）
├── 热路径（每帧调用）→ 考虑转委托
└── 冷路径（初始化/配置）→ 直接用反射就行
```
