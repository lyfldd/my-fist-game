# 04-反射注意事项与坑点

> 学习目标：理解反射中常见的错误、边界情况和使用禁区，避免在实际开发中踩坑。

---

## 坑点一：泛型版本只支持无参构造

### 错误写法

```csharp
public class Monster
{
    public string Name { get; set; }
    public int Hp { get; set; }

    // 只定义了有参构造，没有无参构造
    public Monster(string name, int hp)
    {
        Name = name;
        Hp = hp;
    }
}

// ❌ 报错：Activator.CreateInstance<T>() 调用的是无参构造
Monster m = Activator.CreateInstance<Monster>("lyf", 100);
```

### 原因

| 版本 | 调用的是 | 是否支持传参 |
|---|---|---|
| `CreateInstance<T>()` | 无参构造 `new T()` | ❌ 不支持 |
| `CreateInstance(Type, object[])` | 运行时查找匹配构造 | ✅ 支持 |

当你写了**任何一个有参构造函数**，C# 编译器就**不再自动生成无参构造**，导致泛型版本找不到可用的构造。

### 解决方案

**推荐：同时保留无参和有参构造**

```csharp
public Monster() { }  // ✅ 给反射用无参构造
public Monster(string name, int hp) { ... }  // 正常业务用有参构造
```

这样两种方式都能用：

```csharp
Monster m1 = Activator.CreateInstance<Monster>();                         // ✅ 零开销
Monster m2 = (Monster)Activator.CreateInstance(typeof(Monster), new object[] { "lyf", 100 });  // ✅ 有参
```

## 坑点二：BindingFlags 不完整导致成员"消失"

### 错误写法

```csharp
// ❌ 只查公开实例成员
Type t = typeof(Monster);
t.GetFields();        // 永远只返回公开实例字段
t.GetMethods();       // 永远只返回公开实例方法

// 如果字段是 private 或 static，上面的写法全部找不到！
```

### 原因

`GetFields()`、`GetMethods()` 等方法**默认只返回公开实例成员**，不会返回 private、protected、static。

### 正确写法：显式指定 BindingFlags

```csharp
BindingFlags flags =
    BindingFlags.Public   // ✅ 公开成员
  | BindingFlags.NonPublic  // ✅ 非公开成员（private/protected）
  | BindingFlags.Instance   // ✅ 实例成员
  | BindingFlags.Static;    // ✅ 静态成员

Type t = typeof(Monster);
FieldInfo[] fields = t.GetFields(flags);   // 公开+私有+实例+静态 的所有字段
MethodInfo[] methods = t.GetMethods(flags); // 公开+私有+实例+静态 的所有方法
```

### 常见遗漏

| 想找什么 | 需要的 Flags |
|---|---|
| 私有字段 | `BindingFlags.NonPublic \| BindingFlags.Instance` |
| 静态字段 | `BindingFlags.Public \| BindingFlags.Static` |
| 私有静态方法 | `BindingFlags.NonPublic \| BindingFlags.Static` |
| 所有公开成员 | 默认（不写）|

## 坑点三：值类型装箱拆箱开销

### 装箱发生的场景

```csharp
// 场景 1：值类型字段存入 object[]
Monster m = new Monster();
FieldInfo fi = t.GetField("Hp");  // Hp 是 int（值类型）

int hp = fi.GetValue(m);      // ✅ 返回 object，里面是装箱的 int
int hp2 = (int)fi.GetValue(m); // ✅ 拆箱才能用

// 场景 2：值类型作为 Invoke 参数
int[] args = new int[] { 100 };
methodInfo.Invoke(obj, args);  // ❌ 报错！参数必须是 object[]

object[] boxedArgs = new object[] { 100 };  // ✅ int 装箱进 object[]
methodInfo.Invoke(obj, boxedArgs);

// 场景 3：数组中的值类型
int[] intArray = new int[] { 1, 2, 3 };     // ✅ int[] 本身不装箱
object[] objArray = new object[] { 1, 2, 3 }; // ⚠️ 每个 int 都单独装箱！
```

### 性能影响

| 操作 | 额外开销 |
|---|---|
| `GetValue()` 返回 object（值类型） | 装箱 + 拆箱 = 2次堆分配 |
| `SetValue()` 传入 object（值类型） | 装箱 = 1次堆分配 |
| `Invoke(object[])` 含值类型参数 | 每个值类型都装箱 |
| `int[]` 数组 | ✅ 不装箱，直接连续内存 |

### 优化方案：用泛型委托绕过装箱

```csharp
// 反射 Invoke —— 参数要装箱
Action<object> invokeSpeak = m => methodInfo.Invoke(m, null); // 慢

// Delegate.CreateDelegate + 泛型委托 —— 零装箱
Action speakDelegate = (Action)Delegate.CreateDelegate(typeof(Action), m, method);
speakDelegate();  // 零装箱，和普通方法调用一样快
```

## 坑点四：反射 + 字符串路径 = 类型安全彻底失效

### 危险写法

```csharp
// ❌ 运行时才知道这个类型存不存在
Type t = Type.GetType("练习12.Monster");  // 拼错一个字母，运行时才报错
object m = Activator.CreateInstance(t);

// ❌ 运行时才知道这个方法存不存在
MethodInfo mi = t.GetMethod("Spek");  // 拼错了，运行时才发现
```

### 风险

| 问题 | 后果 |
|---|---|
| 类名拼写错误 | `Type.GetType()` 返回 null，`CreateInstance(null)` 抛异常 |
| 方法名拼写错误 | `GetMethod()` 返回 null，`Invoke(null, ...)` 抛异常 |
| 找不到程序集 | `Type.GetType()` 返回 null |
| 类型转型失败 | `(Monster)obj` 抛 `InvalidCastException` |

### 建议

1. **优先用 `typeof()` 而不是 `Type.GetType()`**
   ```csharp
   // ✅ typeof 编译时检查
   Type t = typeof(Monster);

   // ❌ Type.GetType 运行时才检查
   Type t = Type.GetType("练习12.Monster");  // 危险！
   ```

2. **调用前检查 null**
   ```csharp
   Type t = typeof(Monster);
   MethodInfo mi = t.GetMethod("Speak");
   if (mi == null)
   {
       throw new MissingMethodException("Speak 方法不存在于 Monster 类中");
   }
   mi.Invoke(m, null);
   ```

## 坑点五：私有成员访问的权限问题

### 普通反射无法访问 private 成员

```csharp
public class Monster
{
    private int secretHp;  // private 字段

    private void SecretAttack() { }  // private 方法
}

// ❌ 默认查找找不到
FieldInfo fi = typeof(Monster).GetField("secretHp");     // null
MethodInfo mi = typeof(Monster).GetMethod("SecretAttack"); // null

// ✅ 必须加 BindingFlags.NonPublic
BindingFlags flags = BindingFlags.NonPublic | BindingFlags.Instance;
FieldInfo fi = typeof(Monster).GetField("secretHp", flags);     // ✅ 找到了
MethodInfo mi = typeof(Monster).GetMethod("SecretAttack", flags); // ✅ 找到了
```

### Unity 特殊情况

> ⚠️ 在 Unity 里，即使找到了 private 成员，`Invoke` 也可能失败——因为 Mono 运行时对 `private` 方法有额外安全限制。

## 坑点六：泛型版本的"假反射"陷阱

### 泛型版本 vs 非泛型版本

```csharp
// 泛型版本 —— 编译时优化，真正零开销
Monster m = Activator.CreateInstance<Monster>();
// 编译器直接生成：newobj instance void Monster::.ctor()
// 没有运行时查找，没有装箱，只有普通 new 的开销

// 非泛型版本 —— 真正运行时反射
Monster m = (Monster)Activator.CreateInstance(typeof(Monster));
// CLR 运行时在元数据表里找构造，找到了再 Invoke
// 有运行时查找开销，有装箱
```

### 记住这个区分

| API | 返回值 | 装箱 | 运行时查找 | 适用场景 |
|---|---|---|---|---|
| `CreateInstance<T>()` | `T` | 无 | 无（编译优化） | 无参构造 |
| `CreateInstance(Type, object[])` | `object` | 参数有装箱 | 有 | 有参构造 |
| `MethodInfo.Invoke()` | `object` | 参数有装箱 | 有 | 通用 |
| `Delegate.CreateDelegate()` | 委托类型 | 无 | 有（一次性） | 频繁调用 |

## 坑点七：foreach 无法遍历 object[] 中的值类型

```csharp
int[] values = new int[] { 1, 2, 3 };       // ✅ 值类型数组，连续内存，foreach 效率高
object[] boxed = new object[] { 1, 2, 3 }; // ❌ 每个 int 单独装箱，foreach 时还要拆箱

foreach (int v in values)
{
    Console.WriteLine(v);  // ✅ 快
}

foreach (int v in boxed)   // ⚠️ 每次访问都拆箱
{
    Console.WriteLine(v);  // 慢
}
```

## 总结：避坑检查清单

```
反射使用前，检查：

[ ] 这个类有没有无参构造？（泛型版必检）
[ ] 要找的成员是公开还是私有的？（加 BindingFlags.NonPublic）
[ ] 要找的成员是实例还是静态的？（加 BindingFlags.Static）
[ ] 参数中有值类型吗？（会装箱，考虑优化）
[ ] 类名/方法名有没有拼写错误？（优先用 typeof）
[ ] 找到了吗？（GetXxx 返回 null 的情况要处理）
```

## 常见错误汇总表

| 错误代码 | 错误原因 | 解决方案 |
|---|---|---|
| `CreateInstance<T>(args)` | 泛型版不支持传参 | 换 `CreateInstance(Type, object[])` |
| `GetField("name")` 返回 null | 字段是 private 或 static | 加 `BindingFlags.NonPublic \| BindingFlags.Static` |
| `Invoke()` 报参数错误 | 参数类型不匹配 | 检查 object[] 中每个参数的类型 |
| `Type.GetType("NameSpace.Class")` 返回 null | 程序集名称缺失 | 完整写法 `"NameSpace.Class, AssemblyName"` |
| `(Monster)result` 抛异常 | result 不是 Monster 类型 | 先 `if (result is Monster)` 检查 |
