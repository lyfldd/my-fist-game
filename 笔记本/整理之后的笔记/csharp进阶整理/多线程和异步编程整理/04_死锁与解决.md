# 04_死锁与解决

## 一、什么是死锁？

> **两个线程互相拿着对方想要的锁，谁都不肯放，于是大家一起卡死不动**

生活化比喻：
- 玩家 A 拿着红色钥匙，想要蓝色钥匙
- 玩家 B 拿着蓝色钥匙，想要红色钥匙
- 谁都不让，谁都动不了 → **死锁**

---

## 二、产生死锁的四个必要条件

只要出现这四条，**90% 概率会死锁**：

| 条件 | 说明 |
|---|---|
| **互斥条件** | 锁只能被一个线程持有 |
| **持有并等待** | 拿着锁的同时，还想拿另一个锁 |
| **不可抢占** | 锁不能被强行抢走，只能主动释放 |
| **循环等待** | 形成 A 等 B、B 等 A 的循环 |

---

## 三、容易写出死锁的典型场景

### 场景 1：两个锁 + 不同顺序
**问题**：两个线程按不同顺序获取两把锁

```csharp
// 玩家1：先武器，后防具
void UseWeaponAndArmor()
{
    lock (_weaponLock)
    {
        lock (_armorLock)  // ❌ 嵌套
        {
            // 换装逻辑
        }
    }
}

// 玩家2：先防具，后武器
void UseArmorAndWeapon()
{
    lock (_armorLock)
    {
        lock (_weaponLock)  // ❌ 嵌套，顺序相反
        {
            // 换装逻辑
        }
    }
}
```

**结果**：线程1拿到武器锁，等防具锁；线程2拿到防具锁，等武器锁 → 死锁

---

### 场景 2：背包 + 金币系统互相调用
**问题**：两个独立模块互相调用，各自加锁，形成循环

```csharp
// 买东西：先锁金币，调用背包
void BuyItem()
{
    lock (_goldLock)
    {
        AddItem();  // ❌ 里面又锁背包锁
    }
}

// 卖东西：先锁背包，调用金币
void SellItem()
{
    lock (_backpackLock)
    {
        AddGold();  // ❌ 里面又锁金币锁
    }
}
```

**结果**：A线程买东西：拿金币锁 → 等背包锁；B线程卖东西：拿背包锁 → 等金币锁

> ⚠️ **这是最容易写出死锁的真实场景**：两个系统独立开发，互相调用，嵌套锁就出现了

---

### 场景 3：怪物 AI + 场景管理多层嵌套
```csharp
void MonsterDead()
{
    lock (_monsterLock)
    {
        SceneManager.RemoveMonster(this);  // ❌ 内部又锁场景锁
    }
}

void SceneUpdate()
```

### 场景 4：技能系统 + Buff 系统互相调用
- 技能释放时：锁技能 → 加 Buff → 锁 Buff
- Buff 结束时：锁 Buff → 通知技能 → 锁技能

## 四、死锁的解决方法

### 方法 1：固定加锁顺序（最常用、最有效、必学）

**规则**：所有线程、所有方法，永远按照**相同顺序**去加锁

```csharp
// ✅ 正确：所有方法都统一顺序：先武器，后防具
void UseWeaponAndArmor()
{
    lock (_weaponLock)  // 统一先A
    {
        lock (_armorLock)  // 再B
        {
            // ...
        }
    }
}

void UseArmorAndWeapon()
{
    lock (_weaponLock)  // ✅ 改为先武器
    {
        lock (_armorLock)  // 再防具
        {
            // ...
        }
    }
}
```

---

### 方法 2：不要嵌套锁（最简单、最安全）

**规则**：一个方法里，只加一把锁。绝不写 lock 里面再套 lock

```csharp
// ✅ 好代码：只锁一次，逻辑拆解开
int tmp;
lock (_lockA) { tmp = 变量A; }  // 先取值

// 逻辑处理（不需要锁）

lock (_lockB) { 变量B = tmp; }  // 再赋值
```

**游戏里特别适用**：玩家数据、背包、任务、金币每个系统尽量只锁自己，不互相嵌套调用

---

### 方法 3：减小锁粒度，不要长时间持有锁

**规则**：拿锁的时间越短，死锁概率越低

**不要在锁里做**：
| ❌ 不要做 | 原因 |
|---|---|
| Sleep | 长时间占用锁 |
| 复杂计算 | 延长锁持有时间 |
| 调用别人的模块 | 可能触发嵌套锁 |
| IO、网络、加载 | 等待时间不可控 |

---

### 方法 4：使用 TryLock，超时放弃（高级防御）

```csharp
if (Monitor.TryEnter(_lockA, 10))  // 等10ms
{
    try
    {
        // 拿到锁了，正常执行
    }
    finally
    {
        Monitor.Exit(_lockA);
    }
}
else
{
    // 没拿到，直接放弃，避免死锁
    return;
}
```

---

## 五、实战守则（照着做就行）

| 守则 | 说明 |
|---|---|
| 1. 能不加锁就不加锁 | 先考虑是否真的需要多线程 |
| 2. 要加就只加一把锁 | 减少死锁可能性 |
| 3. 绝不写 lock 里面套 lock | 这是死锁的根源 |
| 4. 必须套锁 → 统一加锁顺序 | 所有地方按相同顺序加锁 |
| 5. 锁里只做赋值，不做复杂逻辑 | 锁越短越好 |
| 6. 锁里不 Sleep、不调用外部方法 | Sleep 占用锁，调用可能嵌套 |

## 六、补充知识点：活锁（Livelock）

### 什么是活锁？
> 两个线程都在**主动**运行，但因为你等我、我等你，**谁都做不了正事**

### 死锁 vs 活锁
| | 死锁 | 活锁 |
|---|---|---|
| 状态 | 线程全部卡住不动 | 线程都在运行，但做无用功 |
| 原因 | 互相等待对方释放锁 | 都在响应对方，永远完不成 |
| 比喻 | 两辆车在窄桥相遇，谁都不退 | 两辆车同时往左让，又同时往右让 |

### 活锁的解决方法
```csharp
// 加入随机等待，打破"同步"节奏
Random rnd = new Random();
Thread.Sleep(rnd.Next(1, 10));  // 随机等1-10ms
```

## 七、补充知识点：读写锁（ReaderWriterLockSlim）

### 适用场景
> 读操作可以并发，写操作需要独占

| 操作 | 规则 |
|---|---|
| 读（多个线程同时读） | ✅ 可以并发 |
| 写（只能一个线程写） | ❌ 必须独占 |

```csharp
private static readonly ReaderWriterLockSlim _rwLock = new ReaderWriterLockSlim();

// 读取（多个线程可同时读）
_rwLock.EnterReadLock();
try
{
    int hp = _hp;
}
finally { _rwLock.ExitReadLock(); }

// 写入（独占）
_rwLock.EnterWriteLock();
try
{
    _hp = 100;
}
finally { _rwLock.ExitWriteLock(); }
```

**游戏场景**：大量读取属性（怪物血量、坐标），少量修改（扣血、移动）

## 八、修正错误汇总表

| 错误认知 | 正确理解 |
|---|---|
| 我小心一点就不会死锁 | ❌ 只要有两个以上的锁+嵌套+顺序不一致，90% 会死锁 |
| 死锁了程序会自己恢复 | ❌ 死锁是永久卡死，必须重启或强制终止线程 |
| 用 TryLock 就不会死锁 | ⚠️ TryLock 只是一种兜底策略，不能从根本上避免死锁 |
| 嵌套锁没问题，只要小心就好 | ❌ 小心不够，核心问题是"不同地方顺序不一致" |
| 活锁比死锁好 | ❌ 活锁线程也在空转，CPU 100%，也很糟糕 |
