# 脚本的执行先后顺序
# 一、概念
当场景中有多个 GameObject，每个 GameObject 上挂载了多个脚本，
这些脚本的生命周期函数（Awake、Start、Update 等）的执行顺序是：
  - 同一类型的生命周期函数，哪个脚本先执行是「不固定的」（默认情况下）
  - 但可以通过设置「脚本执行顺序」来手动控制

理解并控制执行顺序，是解决多个系统之间初始化依赖关系的关键。


# 二、用法
### 方式一：通过 Project Settings 设置执行顺序（推荐）
  菜单栏 → Edit（编辑） → Project Settings（项目设置） → Script Execution Order（脚本执行顺序）
  - 点击 [+] 按钮，添加需要控制顺序的脚本
  - 数字越小，越优先执行（负数也可以）
  - 没有被添加到列表中的脚本按「默认顺序（0）」执行
  - 执行顺序只影响同一生命周期阶段内（如所有脚本的 Update）的相对顺序

### 方式二：使用 [DefaultExecutionOrder] 特性
  在脚本类上方加上特性，直接在代码里声明执行优先级：

  [DefaultExecutionOrder(-100)]   // 负数 = 更早执行
  public class GameManager : MonoBehaviour
  {
```csharp
```
  void Awake() { ... }
  }

  [DefaultExecutionOrder(100)]   // 正数 = 更晚执行
  public class UIManager : MonoBehaviour
  {
```csharp
```
  void Awake() { ... }
  }

  数字越小越先执行，和 Project Settings 里的数字规则一致。


# 三、原理
团结引擎的执行顺序规则：

  阶段一：所有脚本的 Awake（按执行顺序）
  阶段二：所有脚本的 OnEnable（按执行顺序）
  阶段三：所有脚本的 Start（按执行顺序）
  --- 每一帧 ---
  阶段四：所有脚本的 FixedUpdate（按执行顺序，物理步）
  阶段五：所有脚本的 Update（按执行顺序）
  阶段六：所有脚本的 LateUpdate（按执行顺序）

重要规则：
  - 同一阶段的所有脚本执行完毕，才进入下一阶段
  - 比如：所有脚本的 Awake 都跑完，才开始跑所有脚本的 Start
  - 这就是为什么 Start 适合做「跨脚本初始化」——利用阶段差，而不是靠顺序


# 四、坑点
1. 不要依赖"默认顺序"来保证初始化
   → 默认情况下，多个脚本的同一生命周期函数执行顺序是不确定的
   → 如果脚本 A 的 Start 依赖脚本 B 在 Awake 中初始化的数据，

2. 执行顺序设置只在同一阶段内有效
   → [DefaultExecutionOrder(-100)] 只保证这个脚本的 Update 比别的脚本更早执行
   → 不能跨阶段控制（不能让 Update 先于另一个脚本的 LateUpdate）

3. 过度依赖执行顺序是架构问题的信号

# 五、应用场景
- 游戏管理器（GameManager）需要比所有其他脚本更早初始化
  → 设置最低的执行顺序数字（如 -100）

- 相机跟随脚本需要在玩家移动之后执行
  → 相机用 LateUpdate，玩家移动用 Update，天然保证顺序

- 输入系统需要在所有逻辑脚本的 Update 之前先采集输入
  → 给输入脚本设置更小的执行顺序数字


# 六、扩展
- 团结引擎内置系统（物理、动画）有固定的执行位置，
  不受 Script Execution Order 影响：
  物理更新（FixedUpdate） → 动画更新 → LateUpdate

- 如果想彻底控制执行顺序，可以用「事件驱动架构」：
  脚本之间不直接依赖，而是通过事件/委托通知，
  这样就不需要关心哪个脚本先执行了
