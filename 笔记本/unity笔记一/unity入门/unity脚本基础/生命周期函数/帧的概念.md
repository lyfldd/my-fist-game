# 帧的概念
# 一、概念
「帧」（Frame）= 游戏循环的一次完整执行。

游戏的本质是一个「死循环」：
  while (游戏运行中)
  {
```csharp
```
  处理输入
  更新游戏逻辑
  渲染画面
  }

每执行一次这个循环，就叫做「一帧」。
每帧结束时，屏幕上的画面更新一次。


# 二、用法
帧率（FPS，Frames Per Second）= 每秒执行多少帧：
  - 60 FPS = 每秒循环 60 次，每帧约 16.67 毫秒
  - 30 FPS = 每秒循环 30 次，每帧约 33.33 毫秒
  - 帧率越高，画面越流畅

团结引擎中每帧会按顺序执行：
  → FixedUpdate（物理更新，固定间隔，可能一帧执行多次或不执行）
  → Update（逻辑更新，每帧一次）
  → 动画更新（引擎内部）
  → LateUpdate（后处理更新，每帧一次）
  → 渲染

引擎提供了获取帧时间的 API：
  Time.deltaTime         // 上一帧到这一帧经过的时间（秒）
  Time.time              // 游戏启动到当前帧的总时间（秒）
  Time.frameCount        // 当前是第几帧


# 三、原理
团结引擎已经帮我们实现好了游戏主循环（死循环），
我们只需要把逻辑写在对应的生命周期函数里（主要是 Update），
引擎每帧都会自动调用它。

帧率不固定的原因：
  - 每帧要处理的计算量不同（AI、物理、渲染复杂度都会变化）
  - CPU / GPU 性能不同
  - 所以每帧耗时是不固定的，即 Time.deltaTime 每帧都不一样


# 四、坑点
1. 移动/动画速度不能用固定数值，要乘以 Time.deltaTime
   → 如果直接写 transform.position += new Vector3(0.1f, 0, 0)
```csharp
```
 在高帧率下移动快，低帧率下移动慢，帧率不同则速度不同
   → 正确写法：transform.position += new Vector3(speed * Time.deltaTime, 0, 0)
```csharp
```
 这样不管帧率多少，每秒移动的距离都相同

2. 物理逻辑不要写在 Update 里
   → Update 的调用频率随帧率变化，物理计算结果会不稳定
   → 物理相关（刚体移动、施加力）应该写在 FixedUpdate 里
   → FixedUpdate 有固定的时间间隔（默认 0.02 秒），物理计算更稳定

3. 卡帧问题
   → 一帧内计算量过大（大量 AI 计算、复杂物理、GC 垃圾回收）会导致帧率骤降（卡顿）
   → Time.deltaTime 会突然变大，可能导致物体瞬间移动很大一段距离


# 五、应用场景
- Update 里根据 Time.deltaTime 控制移动、旋转速度
- 计时器：在 Update 里累加 Time.deltaTime，到达阈值时触发事件
- 控制动画播放速度
- 基于帧数的逻辑节奏控制


# 六、扩展
- Time.timeScale：控制游戏时间流速
  Time.timeScale = 0   → 游戏暂停（Update/FixedUpdate 都不执行，deltaTime = 0）
  Time.timeScale = 0.5  → 慢动作（时间变为正常的一半）
  Time.timeScale = 2    → 快进

- Application.targetFrameRate：设置目标帧率上限
  Application.targetFrameRate = 60;  // 限制最高 60 FPS

- VSync（垂直同步）：让帧率和显示器刷新率同步，防止画面撕裂
