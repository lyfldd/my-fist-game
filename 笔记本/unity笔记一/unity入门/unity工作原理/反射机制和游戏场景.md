# 反射机制在 Unity（团结引擎）中的应用
# 一、概念
反射（Reflection）是 C# 的一种机制，允许程序在「运行时」：
  - 获取类型的元数据（类名、方法名、字段名等）
  - 动态创建对象实例
  - 动态调用方法、读写字段和属性

团结引擎编辑器本身就是一个"会用反射驱动你代码"的宿主程序，
我们写的脚本被引擎通过反射加载、挂载、调用。

Unity/团结引擎底层大量依赖反射来驱动脚本系统，理解反射是理解 Unity 工作原理的关键。


# 二、用法（团结引擎中反射的三大体现）
1. 脚本挂载 = 通过反射 new 出对象
   当把脚本拖到 GameObject 上时：
   - 引擎用脚本类名，通过反射找到对应的 Type
   - 通过反射 new 出该脚本的实例
   - 把这个实例作为组件（Component）挂到 GameObject 上

   代码动态添加同理：
```csharp
```
 gameObject.AddComponent<MyScript>();

2. 生命周期函数 = 通过反射调用方法
   团结引擎不知道你脚本里写了什么，但它会：
   - 通过反射扫描脚本类，查找固定名字的方法（Awake、Start、Update 等）
   - 在特定时机通过反射调用这些方法
   所以生命周期函数不需要写 public，引擎用反射一样能找到并调用。

3. Inspector 修改变量 = 通过反射读写字段
   在检视窗口（Inspector）修改变量的值时：
   - 引擎通过反射找到该字段
   - 通过反射直接为字段赋新值
   这就是为什么加了 [SerializeField] 的私有变量也能在 Inspector 里被编辑。


# 三、原理
团结引擎启动时会扫描 Assets 下所有 .cs 文件并编译成程序集（Assembly），
然后通过反射：
  1. 收集所有继承 MonoBehaviour 的类型
  2. 记录每个 GameObject 上挂载了哪些脚本（组件）
  3. 在游戏运行的不同阶段，反射调用对应的生命周期函数

本质上，团结引擎 = 一个大型的反射驱动框架，我们的脚本是"被驱动的插件"。

引擎在底层对反射调用做了缓存优化（MethodInfo 缓存），
所以生命周期函数的调用性能影响很小。


# 四、坑点
1. 类名和文件名不一致
   → 旧版 Unity 无法通过反射找到该类，脚本挂载直接失败
   → 团结引擎（Unity 2022+）虽然放宽了这个限制，但仍强烈建议保持一致

2. 手动在 Update 里写大量反射代码性能开销大
   → 引擎底层的反射做了缓存，问题不大
   → 但如果自己在 Update 里频繁手动反射调用，会产生明显性能问题

3. 不是所有类型都能被序列化显示在 Inspector
   → Dictionary 无法直接显示
   → 自定义类/结构体需要加 [System.Serializable] 才能显示
   → 本质原因：团结引擎的序列化系统只支持特定类型

4. 脚本命名空间问题
   → 反射查找类时，命名空间有时会影响查找结果
   → 入门阶段一般不用命名空间，避免出现找不到类的情况


# 五、应用场景
- 理解为什么脚本必须继承 MonoBehaviour 才能挂载到对象上
- 理解 Inspector 面板变量修改的底层原理
- 理解生命周期函数为什么不需要手动调用，引擎会在对应时机自动执行
- 理解"文件名 = 类名"这个规则的来源


# 六、扩展
- 自定义编辑器扩展（Editor 脚本）会大量手动使用反射
- ScriptableObject 的序列化同样依赖反射机制
- Unity/团结引擎的 [Attribute]（特性）系统配合反射使用：
  [Header]、[SerializeField]、[RequireComponent] 等
  都是引擎在用反射扫描脚本时读取特性并执行对应逻辑
- 如果以后想深入研究引擎底层，可以了解 C# 的 Assembly、Type、MethodInfo 等反射 API
