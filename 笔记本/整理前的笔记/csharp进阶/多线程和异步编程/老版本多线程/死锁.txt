死锁到底是什么
一句话：
两个线程互相拿着对方想要的锁，谁都不肯放，于是大家一起卡死不动
玩家 A 拿着红色钥匙，想要蓝色钥匙
玩家 B 拿着蓝色钥匙，想要红色钥匙
谁都不让，谁都动不了
→ 死锁


场景 1：玩家同时使用两件装备，各自上锁
那我们在什么场景下可能会写出这种锁，导致死锁呢
例如
你有两个装备系统：
装备 A：武器（需要锁）
装备 B：防具（需要锁）
代码逻辑很自然：
// 玩家1 使用逻辑：先拿武器，再穿防具
void UseWeaponAndArmor()
{
    lock(武器锁)
    {
        lock(防具锁)
        {
            // 换装逻辑
        }
    }
}

// 玩家2 相反逻辑：先穿防具，再拿武器
void UseArmorAndWeapon()
{
    lock(防具锁)
    {
        lock(武器锁)
        {
            // 换装逻辑
        }
    }
}
线程 1 拿到武器锁
线程 2 拿到防具锁
互相等对方的锁 → 死锁


场景 2：背包 + 金币系统，两个独立模块互相调用
买东西要先扣金币 → 锁金币
扣金币成功后要加物品 → 锁背包
void BuyItem()
{
    lock(金币锁)
    {
        AddItem(); // 里面又 lock(背包锁)
    }
}

void SellItem()
{
    lock(背包锁)
    {
        AddGold(); // 里面又 lock(金币锁)
    }
}
A 线程买东西：拿金币锁 → 等背包锁
B 线程卖东西：拿背包锁 → 等金币锁
这是最容易写出死锁的真实场景
因为两个系统独立开发，互相调用，嵌套锁就出现了。


场景 3：怪物 AI + 场景管理，多层嵌套调用
游戏逻辑：
怪物死亡时：
锁怪物数据
调用场景系统清理尸体
场景系统内部又锁场景
void MonsterDead()
{
    lock(怪物锁)
    {
        SceneManager.RemoveMonster(this);
        // 内部又 lock(场景锁)
    }
}

void SceneUpdate()
{
    lock(场景锁)
    {
        CheckAllMonster();
        // 内部又 lock(怪物锁)
    }
}


场景 4：技能系统 + buff 系统，互相调用
技能释放时：
锁技能
内部加 buff → 锁 buff
buff 结束时：
锁 buff
内部通知技能 → 锁技能
又是互相嵌套 + 顺序相反 → 死锁。

不同模块互相调用，各自加锁，嵌套后形成循环等待
你不是故意写死锁，
你只是：
写 A 模块时，加了锁 A
写 B 模块时，加了锁 B
A 调用 B，B 调用 A
两个线程执行顺序相反
死锁诞生
最容易写出死锁的特征（记住这 3 条）
两个以上的锁
锁嵌套（lock 里面又 lock）
不同地方加锁顺序不一样
只要出现这三条，90% 概率会死锁。
