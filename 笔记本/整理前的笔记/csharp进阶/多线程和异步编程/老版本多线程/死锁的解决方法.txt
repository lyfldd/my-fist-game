方法 1：固定加锁顺序（最常用、最有效、必学）
这是游戏开发、服务器开发解决死锁的首选方案。
规则：
所有线程、所有方法，永远按照相同顺序去加锁。
比如你有三把锁：
锁 A（玩家）
锁 B（背包）
锁 C（金币）
你强制规定一个顺序：
永远先 A → 再 B → 再 C
不管在哪个方法里，只要需要同时加多个锁，必须按这个顺序。

错误（会导致死锁）
// 方法1
lock(B) { lock(A) { } }

// 方法2
lock(A) { lock(B) { } }

正确（不会死锁）
// 所有方法都统一顺序：先A后B
lock(A)
{
    lock(B)
    {
        // ...
    }
}

方法 2：不要嵌套锁（最简单、最安全）
一个方法里，只加一把锁。
绝不写 lock 里面再套 lock

坏代码（嵌套锁）
lock(A)
{
    // ...一些逻辑
    lock(B)  // 嵌套！危险！
    {

    }
}

好代码（只锁一次，逻辑拆解开）
int tmp;
lock(A) { tmp = 变量A; }

// 逻辑处理

lock(B) { 变量B = tmp; }
游戏里特别适用
玩家数据
背包
任务
金币
每个系统尽量只锁自己，不互相嵌套调用。

方法 3：减小锁粒度，不要长时间持有锁
规则：
拿锁的时间越短，死锁概率越低。
不要：
锁里写 Sleep
锁里做复杂计算
锁里调用别人的模块
锁里做 IO、网络、加载
这些都会大幅提高死锁概率。

方法 4：使用 TryLock，超时放弃（高级防御）
如果你实在避免不了嵌套锁，可以用这个保险方案。
C# 里用 Monitor 实现：
if (Monitor.TryEnter(锁A, 10)) // 10ms 超时
{
    try
    {
        // 拿到锁了
    }
    finally
    {
        Monitor.Exit(锁A);
    }
}
else
{
    // 没拿到，直接放弃，避免死循环等待
    return;
}
三、最实用的总结（你以后写代码照着做）
1. 能不加锁就不加锁
2. 要加就只加一把锁
3. 绝不写 lock 里面套 lock
4. 必须套锁 → 统一加锁顺序
5. 锁里只做赋值，不做复杂逻辑
6. 锁里不 Sleep、不调用外部方法
