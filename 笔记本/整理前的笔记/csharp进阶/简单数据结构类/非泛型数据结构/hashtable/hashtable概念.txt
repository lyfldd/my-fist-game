Hashtable 是一种 ** 根据键（Key）直接访问值（Value）** 的数据结构，它不保证排序
就是每一个唯一的键key都有唯一的value与之对应
键值对存储：Key-Value 模式。
Key 不能为空（null），但 Value 可以为空。
Key 必须唯一：不能有重复的 Key，否则会覆盖原值。
非泛型：存储的是 object 类型。
非线程安全：多线程操作需要手动加锁（lock）

底层原理
数组 + 链表：底层是一个数组（桶），每个元素存储的是一个键值对节点
当你存入一个 Key 时，程序会计算这个 Key 的哈希码（Hash Code），通过算法转换成数组的一个索引（位置）
如果两个 Key 算出了同一个索引，Hashtable 会用链地址法（把节点连成一个链表）来存储
底层就是一个数组！不是链表，不是树，就是数组
[ 空 ] [ 空 ] [ 键值对 ] [ 空 ] [ 键值对 ]
把 Key 扔进去 → 算哈希码 → 得到数组下标
Key = "张三"
→ 计算哈希 → 得到下标 3
→ 就把数据存在 array[3]

两个不同的 Key，算出来的下标可能一样！
比如
"张三" → 下标 3
"李四" → 下标 3
同一个下标位置，拉出一条链表
数组[3] → 张三 → 李四 → 王五

 最重要的缺点（为什么现在不用 Hashtable）
非泛型，存 object
→ 装箱拆箱，性能差
类型不安全
→ 取出来要强转
遍历无序
→ 不保证顺序

哈希冲突怎么解决呢
 链地址法
数组的每个位置，不是只存一个元素，而是存一条链表
冲突的元素，直接挂在链表后面
缺点
链表太长 → 查找变慢（变成 O (n)）

 开放寻址法（线性探测）—— 数组里 “找空位”
冲突了？往后找下一个空位置，放进
下标 5 满了 → 看 6 → 6 满了 → 看 7 → 7 空 → 放 7
缺点
容易形成 “聚集”，冲突越来越多
删除非常麻烦（不能直接删，会断链

二次探测法
跳着找
冲突 → 下标+1 → 不行 → 下标+4 → 不行 → 下标+9...
缺点
依然不能随便删
实现复杂

再哈希法
冲突了？换一个哈希函数，重新算一次下标！
缺点
速度慢
计算复杂
