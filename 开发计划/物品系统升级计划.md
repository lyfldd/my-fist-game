## 一、系统概览

物品图谱（ItemGraph）是一个**有向无环图（DAG）**，约 240 个节点，记录游戏中所有物品的完整上下游生产关系。图谱由 ItemGraphBuilder 编辑器工具自动构建，生成 `ItemGraph.asset` ScriptableObject，运行时由 ItemGraphManager 加载并提供查询。

### 1.1 文件清单

| 文件 | 路径 | 职责 |
|------|------|------|
| `ItemGraph.cs` | `_Game/Config/` | ScriptableObject 资产定义，存储节点数组+统计 |
| `ItemGraphNode.cs` | `_Game/Config/` | 单个节点的数据结构（可序列化） |
| `ItemGraphBuilder.cs` | `_Game/Editor/` | EditorWindow：扫描 ItemData + RecipeData，构建 DAG，自动计算深度/工作站/断头路 |
| `ItemGraphInspector.cs` | `_Game/Editor/` | CustomEditor：可视化浏览、搜索过滤、手动覆写 |
| `ItemGraphManager.cs` | `_Game/Systems/ItemGraph/` | 运行时单例，Dictionary 查找，上游/下游/深度/工作台查询 |
| `ChainType.cs` | `_Game/Config/` | 枚举：6 条产业链 |
| `ProductionMode.cs` | `_Game/Config/` | 枚举：手工/工业/双模式 |
| `WorkstationTier.cs` | `_Game/Config/` | 枚举：10 级工作台 |
| `RecipeCatalog.cs` | `_Game/Config/` | ScriptableObject：所有配方的总表 |
| `RecipeData.cs` | `_Game/Config/` | ScriptableObject：单个配方定义 |

### 1.2 资产关系

```
ItemData[]  ─┐
              ├── ItemGraphBuilder ──→ ItemGraph.asset ──→ ItemGraphManager (runtime)
RecipeData[] ┘        │
                      └── 同步 requiredStation → RecipeData[].requiredStation
```

## 二、核心数据结构

### 2.1 ItemGraphNode（可序列化）

```csharp
[Serializable]
public class ItemGraphNode
{
    // 基础信息
    public string itemName;              // 对应 ItemData.itemName
    public ChainType primaryChain;       // 主链类型（自动检测或手动覆写）
    public ProductionMode productionMode; // Manual / Industrial / Both

    // 手动覆写
    public bool overrideStation;         // 勾选后覆盖自动工作台分配
    public WorkstationTier manualStation; // 手动指定的工作台

    // 自动计算字段
    public int[] depths;                 // 所有生产路径的深度集合
    public string[] upstreamItemNames;   // 上游物品（哪些物品是它的材料）
    public string[] downstreamItemNames; // 下游物品（哪些配方消费它）
    public RecipeData[] producerRecipes; // 产出该物品的配方
    public RecipeData[] consumerRecipes; // 消费该物品的配方
    public int consumerCount;            // 下游配方数量 = 消费热度
    public WorkstationTier autoAssignedStation; // 规则自动分配的工作台
    public bool isRawMaterial;           // 无配方产出它（只能从世界获取）
    public bool isDeadEnd;              // 无配方消费它（断头路）

    // 功能实现追踪
    public string[] requiredSystems;     // 依赖的系统名称（从 ItemData.behaviours 汇总）
    public bool allSystemsReady;         // 所有依赖系统的脚本是否都已实现

    // 计算属性
    public WorkstationTier EffectiveStation  // overrideStation ? manualStation : autoAssignedStation
    public int MinDepth                       // depths 的最小值
    public int MaxDepth                       // depths 的最大值
}
```

### 2.2 ItemGraph（ScriptableObject 资产）

```csharp
public class ItemGraph : ScriptableObject
{
    public string buildTimestamp;       // 构建时间戳
    public int recipeCountAtBuild;      // 构建时的配方总数
    public ItemGraphNode[] nodes;       // 所有物品节点（约 240 个）
    public int rawMaterialCount;        // 原材料数量
    public int deadEndCount;            // 断头路数量
    public int coreMaterialCount;       // 核心材料数量（消费热度 >= 10）
}
```

## 三、DAG 构建流程（ItemGraphBuilder）

### 3.1 构建入口

菜单路径：**Tools → 物品图谱 → 构建 ItemGraph**

`ItemGraphBuilder.Build()` 是 EditorWindow 的静态方法，执行以下 9 步：

### 3.2 步骤详解

#### Step 1 — 收集 ItemData

```csharp
AssetDatabase.FindAssets("t:ItemData")
```
扫描项目中所有 `ItemData` ScriptableObject，提取 `itemName` 加入 `allItemNames` HashSet。

#### Step 2 — 收集 RecipeData

```csharp
AssetDatabase.FindAssets("t:RecipeData")
```
扫描所有 `RecipeData`。只保留 `resultItem != null` 的有效配方。

#### Step 3 — 构建邻接表

建立两个核心映射：
- `upstreamMap[outputName]` = 产出该物品所需的所有输入物品名（**它由什么做成**）
- `downstreamMap[inputName]` = 消费该材料的所有输出物品名（**它被谁用**）
- `recipeCountForOutput[itemName]` = 产出该物品的配方数（含手工+工业）

遍历每个配方的 `materials[]`，建立双向关系：
```
配方: 铁锭 + 煤炭 → 钢锭
结果: upstreamMap["钢锭"].Add("铁锭"), upstreamMap["钢锭"].Add("煤炭")
      downstreamMap["铁锭"].Add("钢锭"), downstreamMap["煤炭"].Add("钢锭")
```

#### Step 4 — 原材料识别

```csharp
rawMaterials = { name | recipeCountForOutput[name] == 0 }
```
无任何配方产出该物品 → 它只能从世界获取（采集/掉落/购买）。

#### Step 5 — BFS 深度计算

核心算法：**迭代 BFS + 多路径深度记录**。

```csharp
depths[itemName] = HashSet<int>  // 每条生产路径的深度都记录
```

- **原材料**深度 = `{0}`
- **无材料配方**深度 = `{1}`（凭空手搓，如徒手配方）
- **有材料配方**深度 = `max(所有材料的 MinDepth) + 1`

BFS 迭代直到稳定（最多 50 轮）。每轮遍历 `recipesByOutput`（产出→配方列表），当配方所有材料都有深度时，计算新深度并加入对应物品的 depth 集合。

**循环依赖打破**：如果 BFS 收敛后仍有物品 depth 为空：
1. 无配方产出的物品 → 强制 depth={0}，归类为原材料
2. 有配方产出的物品 → 取**材料最少的配方**，用已知材料深度 +1，忽略未知材料
3. 重新传播一轮（最多 30 轮）

#### Step 5.5 — 配方引用查找表

构建 `producerRecipes[name]`（产出该物品的配方列表）和 `consumerRecipes[name]`（消费该物品的配方列表），挂载到每个节点。

#### Step 6 — 构建 ItemGraphNode 数组

对每个物品名，计算并填充所有字段：
- 深度数组（去重排序）
- 上/下游物品名（去重排序）
- 链类型（自动检测，见第四节）
- 自动工作台分配（见第四节）
- 工业设备等级覆盖（取 autoStation 与工业配方 requiredStation 的最大值）
- isRawMaterial / isDeadEnd
- requiredSystems / allSystemsReady（从 ItemData.behaviours 提取）
- 消费热度 = downstreamArr.Length

#### Step 7 — 保存 ItemGraph.asset

```csharp
const string savePath = "Assets/_Game/Config/ItemGraph.asset";
AssetDatabase.CreateAsset(graph, savePath);
```

#### Step 8 — 同步 RecipeData.requiredStation

遍历所有 RecipeData，将每个配方产出物品的 `EffectiveStation` 写回配方的 `requiredStation`。如果有差异，用 `Undo.RecordObject` 记录并 `SetDirty`。

#### Step 9 — 输出诊断报告

Console 输出 4 段报告：
- 断头路清单（排除原材料，按热度排序）
- 核心材料（热度 >= 5，按热度排序，含 depth/station）
- 工作台分配统计（每个工作台的物品数）
- 链类型统计（每条链的物品数）

### 4.3 手动覆写

在 ItemGraphInspector 中可以手动覆写节点的：
- 链类型（覆盖关键词检测结果）
- 生产模式（覆盖默认 Behavior 推理）
- 工作台（勾选 `overrideStation` 后覆盖自动分配）

覆写后重新构建图谱不会覆盖手动设置（因为覆写信息存在 ItemGraph.asset 中）。

## 五、运行时查询（ItemGraphManager）

### 5.1 初始化

`ItemGraphManager` 是 DontDestroyOnLoad 单例，在 `Awake()` 中：
1. 加载 `Resources.Load<ItemGraph>("ItemGraph")`
2. 加载 `Resources.Load<RecipeCatalog>("RecipeCatalog")`
3. 调用 `BuildLookup()` 构建 `Dictionary<string, ItemGraphNode>` 加速查询

### 5.2 查询接口

| 方法 | 返回 | 用途 |
|------|------|------|
| `GetNode(name)` | `ItemGraphNode` | 获取完整节点（null = 不在图谱中） |
| `GetDownstream(name)` | `string[]` | 哪些配方消费它 |
| `GetUpstream(name)` | `string[]` | 它由哪些材料做成 |
| `GetMinDepth(name)` | `int` | 最浅生产深度 |
| `IsDeadEnd(name)` | `bool` | 是否断头路 |
| `IsRawMaterial(name)` | `bool` | 是否原材料 |
| `GetEffectiveStation(name)` | `WorkstationTier` | 生效的工作台等级 |
| `GetAllDeadEnds()` | `string[]` | 所有断头路物品名 |
| `GetCoreMaterials()` | `string[]` | 所有核心材料（热度 >= 10） |
| `GetRecipesForStation(tier)` | `RecipeData[]` | 指定工作台的所有手工配方 |
| `GetRecipesForDevice(name)` | `RecipeData[]` | 指定设备的全部工业配方 |
| `GetRecipesForItem(name)` | `RecipeData[]` | 产出该物品的所有配方（手工+工业） |
| `GetDevicesForItem(name)` | `string[]` | 可以工业量产该物品的设备名列表 |

### 5.3 使用场景

- **合成面板**：`GetRecipesForStation(tier)` 获取当前工作台可做的所有配方
- **工业设备面板**：`GetRecipesForDevice(name)` 获取指定设备的量产配方
- **物品提示**：`GetUpstream(name)` + `GetDownstream(name)` 显示物品的生产链
- **存档/经济平衡**：`GetCoreMaterials()` 识别热门材料，监控经济循环
- **设计审查**：`GetAllDeadEnds()` 发现无下游消费的物品，及时补配方

## 六、编辑器可视化（ItemGraphInspector）

### 6.1 功能

打开 `ItemGraph.asset` 即可看到 CustomEditor 界面，提供：

**统计条**：
```
物品 240 | 原材 45 | 断头 12 | 核心 18 | 缺脚本 7 | 2026-06-22 15:30:00
```

**过滤器**：
- 搜索：物品名模糊匹配
- 链类型：Metal / Electronics / Chemical / Biological / Food / Energy
- 断头路 / 核心材料 / 原材料 / 缺脚本：开关过滤
- 深度范围：minDepth ~ maxDepth 区间

**颜色编码**：
- 🔴 暗红底 = 断头路（非原材料）
- 🟢 深绿底 = 核心材料（热度 >= 10）
- 🟠 橙黄底 = 缺脚本（allSystemsReady = false）
- 🔵 蓝底 = 当前选中

**列表行格式**：
```
[金] 钢锭          d=1  ↓18  Furnace
[电] 电路板         d=3  ↓5   AdvancedBench   ⚠
[化] 煤焦油          d=2  X    Chemistry
```

- `[金]` = 链类型简称（金/电/化/生/食/能）
- `d=1` = MinDepth
- `↓18` = 消费热度（下游配方数），`X` = 断头路，`R` = 原材料
- `Furnace` = EffectiveStation
- `⚠` = 缺脚本
- `[覆]` = 手动覆写工作站

**详情面板**（选中节点后显示）：
- 完整统计：链类型、深度集合、消费热度
- 原材料/断头路标记
- 自动工作站 / 生效工作站
- 系统就绪状态（依赖系统列表 + allSystemsReady）
- 产出配方列表（含 [手工]/[工业] 标签 + 材料清单）
- 消费配方列表（含标签 + 产出物）
- 上游物品列表（含 depth + chain）
- 下游物品列表（含 depth + chain）
- 手动覆写编辑区（覆写开关、工作站、链类型、生产模式）

### 6.2 手动覆写

在详情面板底部可以：
- 勾选"覆写工作站"并指定等级
- 修改链类型
- 修改生产模式
- **保存修改**按钮：用 `Undo.RecordObject` 写入 ItemGraph.asset

当链类型变更时，`autoAssignedStation` 会根据新链类型重新计算。

## 七、行为追踪系统

### 7.1 追踪机制

`ItemGraphNode` 记录每个物品的**脚本实现状态**：

- `requiredSystems: string[]` — 从 `ItemData.behaviours` 汇总，提取 `b.system.ToString()` 去重
- `allSystemsReady: bool` — 当所有 behaviour 的 `componentTypeName` 非空且 `implemented = true` 时为 true

### 7.2 判断逻辑

```csharp
foreach (var b in itemData.behaviours)
{
    requiredSystemsList.Add(b.system.ToString());        // 系统名去重
    if (!string.IsNullOrEmpty(b.componentTypeName) && !b.implemented)
        allReady = false;                                 // 有组件名但未实现
}
```

### 7.3 可视化

- 列表行：`allSystemsReady = false` 的节点显示 `⚠` 且为橙黄底色
- 详情面板：
  - 就绪：显示 `✅ 系统就绪` + 依赖列表 或 `(无特殊脚本需求)`
  - 未就绪：显示 `⚠ 缺脚本` + 依赖系统列表
- 过滤器：可单独筛选 `缺脚本` 的节点

## 八、DAG 统计与诊断

### 8.1 统计指标

| 指标 | 来源 | 含义 |
|------|------|------|
| 总物品数 | `nodes.Length` | DAG 中的节点总数（约 240） |
| 原材料数 | `rawMaterialCount` | recipeCountForOutput == 0 |
| 断头路数 | `deadEndCount` | downstreamItemNames.Length == 0 |
| 核心材料数 | `coreMaterialCount` | consumerCount >= 10 |
| 缺脚本数 | `nodes.Where(!allSystemsReady).Count()` | 功能未实现的物品 |
| 构建时间 | `buildTimestamp` | 最后一次 Build() 时间 |

### 8.2 断头路检测

断头路定义：有配方产出该物品，但没有任何配方消费它（`isDeadEnd && !isRawMaterial`）。

这意味着该物品是生产的终点——制造出来后就无处可去。设计上需要：
- 断头路如果是**终端消费品**（如最终武器、建筑套件）→ 合理
- 断头路如果是**半成品**（如中间材料）→ 需要添加下游配方

### 8.3 核心材料识别

消费热度 >= 10 的物品被标记为核心材料。这些是产业链中的"热门中间件"：
- 需要充足的世界产出或稳定的工业量产
- 在存档经济中应重点监控库存
- 是产业链设计中的关键节点

## 九、与配方系统的集成

### 9.1 构建时同步

ItemGraphBuilder 的 Step 8 会遍历所有 RecipeData，将图谱计算的 `EffectiveStation` 同步到配方的 `requiredStation`：

```
RecipeData.requiredStation = ItemGraphNode.EffectiveStation
```

同步时使用 `Undo.RecordObject` 保证可撤销，并在 Console 输出每个变更。

### 9.2 运行时集成

- **合成 UI** 通过 `ItemGraphManager.GetRecipesForStation(tier)` 动态获取配方列表
- **工业设备** 通过 `GetRecipesForDevice(name)` 获取量产配方
- **配方解锁** 可查询 `GetRecipesForItem(name)` 找到所有产出某物品的配方

## 十一、边界情况与设计决策

### 11.1 循环依赖

如果存在 A→B→A 的配方循环，BFS 会无法收敛。ItemGraphBuilder 的循环打破机制：
1. 无配方产出的物品强制 depth=0
2. 有配方但深度未解决的，取材料最少的配方、忽略未知材料计算近似深度
3. 重新传播一轮

### 11.2 不在图谱中的物品

运行时 `GetNode(name)` 返回 null。调用方应自行处理（通常跳过或提示"未纳入产业链"）。RecipeCatalog 同步时，不在图谱中的产出物品会输出 Warning。

### 11.3 同名物品

系统假设 `ItemData.itemName` 唯一。如果有重名，后者会覆盖前者的节点。

### 11.4 多配方产出同一物品

一个物品可以有多个产出配方（手工 + 工业 + 多种配方）。BFS 会取所有路径中**最浅的深度**作为工作台分配依据（因为最浅路径通常是手工路径，更早解锁）。
