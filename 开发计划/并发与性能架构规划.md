### 一、核心认知：三个瓶颈，三种解法

```
╔══════════════════════════════════════════════════════╗
║                     你的游戏循环                       ║
╠══════════════════════════════════════════════════════╣
║  CPU 逻辑层 ← 你的代码在这里                           ║
║  ┌─────────────┬─────────────┬─────────────────────┐ ║
║  │ AI 状态机    │ 寻路/感知    │ 背包/合成/建造      │ ║
║  │ (状态切换)   │ (NavMesh)  │ (数据操作)          │ ║
║  └──────┬──────┴──────┬──────┴──────────┬──────────┘ ║
║         │             │                 │            ║
║         ▼             ▼                 ▼            ║
║  PhysX 物理引擎 (C++ 层，已内置多线程)                  ║
║         │                                            ║
║         ▼                                            ║
║  GPU 渲染管线 (SRP Batcher + GPU Instancing)          ║
╚══════════════════════════════════════════════════════╝
```

**渲染瓶颈** = GPU 问题 → 不靠 CPU 线程解决
**物理瓶颈** = PhysX 已多线程 → 你的代码只是在读结果
**AI/逻辑瓶颈** = 真正的 CPU 瓶颈 → 这才是多线程/JOB 的主战场

**结论：不要用 C# 裸多线程（`Task.Run`/`new Thread`），用 Unity Job System + Burst Compiler。**

| 方案 | 真正多核并行 | 安全访问 Unity API | GC 压力 | 适用场景 |
|------|:---:|:---:|:---:|------|
| **协程** | ❌ 仍跑主线程 | ✅ | ❌ 有分配 | 延时、逐帧分散计算量 |
| **C# 裸多线程** | ✅ | ❌ 极易崩溃 | ❌ Task 分配多 | ~~游戏中基本无适用场景~~ |
| **Job System + Burst** | ✅ | ✅ (NativeContainer + Safety Checks) | ✅ (值类型零 GC) | 计算密集：噪声、寻路批处理、AI 批量决策 |
| **ECS + Jobs** | ✅ | ✅ (Entities Graphics) | ✅ | 500+ 实体极致性能 |

### 三、架构设计铁律（从现在开始遵守）

#### 铁律 1：决策与执行分离

```csharp
// ❌ 现在不这样写（决策逻辑和 Unity API 混在一起）
void Update()
{
    float dist = Vector3.Distance(transform.position, target.position);  // 决策
    if (dist < attackRange)
        agent.SetDestination(target.position);  // Unity API
}

// ✅ 以后这样设计
struct AIDecisionInput   // 纯数据 struct，可塞进 Job
{
    public Vector3 position;
    public Vector3 targetPosition;
    public float attackRange;
    // ... 所有决策需要的输入
}

struct AIDecisionOutput  // 纯数据 struct
{
    public bool shouldChase;
    public Vector3 destination;
    public AIState newState;
}

class AIDecisionSystem   // 可以 Job 化
{
    public AIDecisionOutput Decide(AIDecisionInput input) { ... }
}

class AIExecutionSystem  // 必须主线程
{
    public void Execute(AIDecisionOutput output, NavMeshAgent agent)
    {
        agent.SetDestination(output.destination);  // Unity API 集中调用
    }
}
```

**这不是现在要重构的代码，而是从现在开始新写代码时脑子里要有这根弦。**

#### 铁律 2：感知数据集中管理，不分散查询

你的 `AIAgent.PerceptionTick()` 已经在做 `Physics.OverlapSphere` → `ThreatSystem.Instance.AddThreat` 的模式，方向是对的。

进一步的方向：不只是每个 AI 自己做感知，而是由一个 **集中式感知系统** 统一跑检测，结果写入一个 NativeArray，所有 AI 直接读——这样感知可以 Job 化，感知结果可以被多个系统复用。

#### 铁律 3：纯数据不用 MonoBehaviour

```csharp

public class AIAgentData : ScriptableObject { ... }

// ❌ 不要每个数据容器都套 MonoBehaviour
public class AIAgentDataMB : MonoBehaviour { ... }  // 没必要
```

你已经在用 `AIAgentData : ScriptableObject`（`AIAgent.cs:37`），继续保持。

### 五、里程碑判断标准

| 触发条件 | 行动 |
|----------|------|
| AI 僵尸/队友 **< 50 个** | 全部主线程 Update，不做任何优化 |
| AI 实体 **50-200 个** | 开启时间分片（当前 `_tickGroup` 模式） |
| **Profiler 显示 `AIAgent.Update` 占比 > 30%** | 引入 Hybrid Jobs（决策 Job 化，执行留主线程） |
| AI 实体 **> 200 且仍有卡顿** | 考虑 ECS 迁移（⚠️ 大工程，评估是否真有必要） |
| **场景加载/卸载导致明显卡顿** | WorldGen 用 Job 异步生成 Chunk |
| **DrawCall > 500 或 Batches > 200** | GPU Instancing + Occlusion Culling Bake |
| **物理耗时 > 5ms/帧** | 物理层 Profile → 考虑 DOTS Physics |

### 六、关键决策：这游戏大概率不需要 ECS

Project Zomboid 几千个僵尸同时存在，靠的是：
- 僵尸用纯数据 + 极少 Unity API 调用
- 远处僵尸几秒才更新一次（LOD 逻辑化）
- 渲染用自研 Java/LWJGL 引擎
- **没有用 ECS**

我们的游戏体量 = Project Zomboid 的缩略版，**时间分片 + Hybrid Jobs 大概率是最终的方案**。ECS 留作"万一"的退路，不做目标。
