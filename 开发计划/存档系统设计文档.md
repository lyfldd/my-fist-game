## 概述

### 这是一个什么游戏？

一款 **3D + 45° 俯视角僵尸生存游戏**，玩法参照《Project Zomboid》。核心循环是：在程序生成的开放城市中搜集物资、建造据点、驾驶车辆、对抗僵尸，尽可能活下去。

### 现在是什么状态？

项目已开发到**阶段 3（内容填充）**，以下系统全部就绪：

```
✅ 生存系统（生命/饥饿/口渴/体温/出血/感染/骨折）
✅ 多容器背包（7 个容器：上衣口袋/裤子口袋/腰带/胸挂/背包/头盔/防弹衣）
✅ 装备系统（护甲/保暖/武器槽/小刀位/手枪位）
✅ 武器系统（射击/换弹/弹夹管理/散射锥）
✅ 技能系统（4 属性 + 10 技能 + 经验池兑换升级）
✅ 建造系统（墙壁/地板/家具 — 放置+拆除）
✅ 世界生成（程序化城市地图 — seed → 街区 → 建筑 → 道路）
✅ 世界容器（房屋内搜刮容器 — 每个容器独立状态）
✅ 车辆系统（驾驶/燃料/上下车）
✅ 时间系统（24h 日夜循环, DayChanged 事件）
✅ 天气系统（马尔可夫链: Sunny/Cloudy/Rain/Storm + 环境温度 + 光照）
✅ 阵营系统（Player/Zombie/Survivor/Bandit/Military/AIBot — 6阵营关系网）
```

但所有这些状态——玩家的血量、背包里的斧头和罐头、刚建好的墙壁、搜刮过的冰箱——**全部只存在于内存里**。关掉游戏，一切归零。

### 我们要做什么？

**从零构建完整的存档系统。**不是简单的"存个位置然后读回来"——是要让玩家关掉游戏、明天打开、发现**一切都在原地等她**。

具体来说，要存这些东西：

| 类别 | 存什么 | 举个例子 |
|------|--------|---------|
| 🧍 玩家状态 | 位置、血量、饥饿口渴、技能等级、体力 | 上次在厨房关的游戏，读档还在厨房 |
| 🎒 背包装备 | 7 个容器里的所有物品 + 身上穿的装备 + 武器弹药 | 背包格子里那把耐久 30% 的斧头，回来还是那把 |
| 🏗️ 世界建筑 | 玩家放置的所有墙壁/地板/家具 | 花了两小时建的据点，一面墙都不少 |
| 📦 世界容器 | 所有搜刮过的容器状态(打开/未打开 + 内容物) | 冰箱里那罐吃了一半的豆子还在 |
| 🤖 系统实体 | AI 机器人、生产设备、车辆、电力网 | 停在门口的皮卡，油箱还是满的 |
| 🧟 附近僵尸 | 玩家周围 30m 内的僵尸(位置/血量/仇恨) | **不能让你靠读档刷掉追你的僵尸群** |
| 🌍 全局状态 | 游戏天数、天气、科技研究、阵营关系 | 第 12 天的暴雨中，回来还是暴雨 |

### 怎么做？

**五个核心设计决策：**

1. **格式用纯 JSON**（不用 SQLite）— 500KB 的存档，记事本就能打开调试。存档数量不会大到需要 SQL 的程度
2. **给每个"东西"发身份证**（PersistentGUID）— 每面墙、每辆车、每个机器人都有一个永不重复的 UUID。Unity 自带的 InstanceID 每次重开游戏都会变，不能用来"认出"上次那个建筑
3. **世界不完整重存** — 城市由 seed 程序生成，只存玩家**改动过的**东西（放置的建筑、丢在地上的物品）。不改动的不存，磁盘占用极小
4. **物品用名字做键** — 不存 `ItemData` 的 Unity 内部引用（序列化不了），而是存 `itemName` 字符串，加载时通过 ItemCatalog 查表找回。建筑同理用 `displayName`
5. **序列化放后台线程** — 主线程只负责把各系统的数据"采集"到内存对象中（<5ms），JSON 转换和写入磁盘扔到后台线程做。玩家感觉不到任何卡顿

**分五个阶段，由浅入深：**


## 3. 技术选型

### 3.1 为什么是纯 JSON 而不是 SQLite？

| 维度 | 纯 JSON（选中） | SQLite |
|------|:--:|:--:|
| 数据量 | ~200-500KB/档，JSON 轻松处理 | 杀鸡用牛刀 |
| 可调试性 | 记事本直接打开 | 需要 SQL 工具 |
| 依赖 | Unity 官方 NuGet 包 | 第三方库，团结引擎兼容性未知 |
| 崩溃安全 | 原子 rename（POSIX 保证） | WAL 模式，同样安全 |
| 维护成本 | 零 | Schema 迁移需要写 SQL 脚本 |

**结论**: 单机 PC 游戏 + 5 个存档槽，纯 JSON 完全够用。SQLite 是过度工程化。

### 3.2 JSON 性能验证

```
存档大小估算:
- 玩家状态:      ~5KB
- 背包物品:      ~10KB (100格)
- 世界建筑:      ~50KB (500个建筑)
- 地面物品:      ~20KB (200个掉落物)
- 容器:          ~30KB (200个容器)
- 系统实体:      ~15KB (AIBot/电力/车辆)
- 附近僵尸:      ~5KB  (30m内 ≤30只)
- 全局状态:      ~2KB
───────────────────────────
总计:           ~135-530KB

序列化: ~50ms (Newtonsoft.Json, 后台线程)
磁盘写入: <20ms (SSD)
玩家感知: 无（数据采集在主线程 <5ms，其余全异步）
```

### 4.2 磁盘文件结构

```
Application.persistentDataPath/Saves/
├── Slot_0/
│   ├── save.json          # 全量 JSON (~200-500KB)
│   ├── save.json.backup   # 上一版备份
│   ├── meta.json          # 元数据 {version, date, playTime, playerLevel, ...}
│   └── thumb.png          # 截图缩略图
├── Slot_1/
│   └── ...
├── Slot_2/
├── Slot_3/
└── Slot_4/
```

**原子写入流程**（全程 POSIX rename，零复制风险）:

```
步骤0: SaveLoadManager.Awake() 扫描并删除所有 *.tmp 残留文件（上次崩溃遗留）
步骤1: 序列化 → 写入 save.json.tmp （写临时文件，不影响正式档）
步骤2: File.Move(save.json → save.json.backup) （旧档直接改名。若崩溃在此步之前，正式档完整；在此步之后，backup完整）
步骤3: File.Move(save.json.tmp → save.json)     （新档就位）
```

> ⚠️ **为什么不用 Copy + Move？** `File.Copy` 不是原子的。如果系统在 Copy 中途崩溃，backup 文件将是**半截文件**，无法用于恢复。
> `File.Move` 在同分区内是纯 rename（只修改文件系统目录项，不动数据块），操作系统保证其原子性——要么成功、要么没发生，绝不存在中间态。
>
> **.tmp 清理**: 步骤 1 之后、步骤 2 之前崩溃会在磁盘留下 `save.json.tmp` 残留。`SaveLoadManager.Awake()` 在初始化时扫描所有槽位的 `*.tmp` 文件并删除，防止累积占用磁盘。

### 4.3 存档槽元数据 (SaveSlotMeta)

```csharp
/// <summary>
```

**死亡存档行为**:
- 玩家死亡时自动保存，`isDead = true`
- UI 显示为灰色 + "已死亡 — 第 X 天"

## 5. 关键设计——PersistentGUID

这是整个存档系统最重要的创新。

### 5.1 问题

当前代码库中大量用 `GameObject.GetInstanceID()` 来追踪实体（如 `EntityDeathEvent`）。但 InstanceID 每次进入 Play Mode 都会重新分配——**跨会话不保证一致**。加载存档时无法用 InstanceID 找回"上次那个建筑"。

### 5.2 方案

```csharp
/// <summary>
/// 贴在需要跨会话追踪的 GameObject 上。
/// GUID 在 Awake() 生成并 [SerializeField] 持久化到场景/prefab，编译重载后不变。
/// </summary>
public class PersistentGUID : MonoBehaviour
{
    [SerializeField] private string guid;       // UUID v4, 首次 Awake 生成
    [SerializeField] private string entityType; // "PlacedStructure"/"AIBot"/"Vehicle"/...
    private bool _initialized;

    void Awake()
    {
        if (string.IsNullOrEmpty(guid))
            guid = System.Guid.NewGuid().ToString("N");
    }

    void Start()
    {
        // 延迟注册，给加载系统覆盖 GUID 的窗口
        if (!_initialized)
        {
            PersistentGUIDRegistry.Register(guid, gameObject);
            _initialized = true;
        }
    }

    /// <summary>加载存档时调用，覆盖 GUID 并注册</summary>
    public void Initialize(string overrideGuid, string type)
    {
        // ⚠️ 关键：先注销 Awake/Start 可能已自动注册的旧 GUID，防止 Registry 残留脏条目
        if (!string.IsNullOrEmpty(guid))
            PersistentGUIDRegistry.Unregister(guid);
        guid = overrideGuid;
        entityType = type;
        PersistentGUIDRegistry.Register(guid, gameObject);
        _initialized = true;
    }
}

/// <summary>
/// 全局单例。O(1) 字典查找 GUID → GameObject。
/// </summary>
public static class PersistentGUIDRegistry
{
    private static Dictionary<string, GameObject> _lookup = new();

    public static void Register(string guid, GameObject go) => _lookup[guid] = go;
    public static void Unregister(string guid) => _lookup.Remove(guid);
    public static GameObject Find(string guid) => _lookup.GetValueOrDefault(guid);
    public static bool Exists(string guid) => _lookup.ContainsKey(guid);
    public static void Clear() => _lookup.Clear();
}
```

### 5.3 哪些实体需要 PersistentGUID？

| 需要 ✅ | 不需要 ❌ (原因) |
|---------|------|
| PlacedStructure（建造系统产出） | WorldContainer（已有稳定的 `int containerId`） |
| PowerSource / PowerTerminal / PowerConsumer | 纯环境装饰物 |
| ProductionDevice | 玩家 GameObject（只有一个，无需追踪） |
| AIBot | 地面物品 WorldItem（临时实体，拾取即消失） |
| Vehicle | |
| Zombie（需要保存的） | |

### 5.4 关键陷阱：加载时 GUID 覆盖 + 注册竞态

**问题**: prefab 无法预存 GUID。同一个建筑 prefab 拖进场景 N 个实例，每个在 Awake 时各自生成 GUID。但加载存档时 `Instantiate(prefab)` 也会生成**新**的 GUID，不会保留存档中的 GUID。

**竞态风险**: Unity 不保证 `Instantiate()` 后 `Start()` 在 `Initialize()` 之后执行。如果 `Start()` 先于 `Initialize()` 触发，会用 Awake 自动生成的旧 GUID 注册到 Registry。后续 `Initialize()` 虽然覆盖了 `guid` 字段，但 Registry 中仍残留旧 GUID → 重复 GUID 的脏条目。

**解决 —— `Initialize()` 内先 Unregister 再 Register**（见 §5.2 代码中的关键注释）:

```csharp
// 加载建筑时
GameObject buildingGO = Instantiate(buildingPrefab, saveData.position, saveData.rotation);
// Awake() 已执行 → 生成了随机新 GUID
// Start() 可能已触发自动注册（如果 Start 先于 Initialize 执行）
buildingGO.GetComponent<PersistentGUID>().Initialize(saveData.guid, "PlacedStructure");
// Initialize() 内部: Unregister(旧GUID) → Register(存档GUID) → 干净 ✅
```

**规则**: 所有实体组件在加载场景时，**不依赖 Awake/Start 自动注册**，而是由加载流程显式调用 `Initialize()`。`Initialize()` 必须**先 Unregister 再 Register**，确保无论 Awake/Start 和 Initialize 的执行顺序如何，Registry 中都是正确的 GUID。

## 6. 数据模型——存什么

### 6.1 顶层 SaveData

```csharp
[Serializable]
public class SaveData
{
    public int version = 1;                    // 存档格式版本号（迁移用）
    public int worldGenVersion = 1;            // 世界生成算法版本号（算法更新后递增，加载时校验）
    public int worldSeed;                      // 世界生成种子
    public string saveDateTime;                // ISO 8601 保存时间
    public float totalPlayTime;                // 累计游戏时间（秒）

    public PlayerSaveData player;
    public InventorySaveData inventory;
    public WorldSaveData world;
    public PowerSaveData power;
    public List<AIBotSaveData> aiBots;
    public List<ProductionSaveData> productions;
    public List<VehicleSaveData> vehicles;
    public List<ZombieSaveData> nearbyZombies;     // 半径 30m 内的僵尸
    public TimeWeatherSaveData timeWeather;
    public ResearchSaveData research;
}
```

> ⚠️ **worldGenVersion**: 世界由 seed 程序生成。当世界生成算法更新后（调整街区布局、建筑分布等），同一 seed 可能产生不同世界。此时存档中建筑的坐标、containerId 将错位。`worldGenVersion` 在加载时与当前算法版本比对，不匹配则拒绝加载并提示"存档世界版本不兼容，请开始新游戏"。

**为什么用 `instanceId` 而不是 `slotIndex`？**

```
❌ slotIndex 方案：
  假设装备的是 placedItems[1]（耐久30%的斧头）。
  但 RestoreFromSave() 恢复物品时，列表插入顺序可能与存档时不同
  （大物品先插入/容器网格重排/堆叠逻辑差异）
  → placedItems[1] 变成了耐久100%那把 → 装备错误

✅ instanceId 方案：
  PlacedItem 创建时分配唯一递增 int instanceId
  存盘: equippedItems["RightHand"] = { itemInstanceId: 42 }
  读档: 遍历所有容器的所有 placedItems，找到 instanceId==42 的那个 → 装备
  → 不依赖列表顺序，100% 定位到同一物品
```

**instanceId 实现**:
- `PlacedItem` 新增 `public int instanceId` 字段
- `Inventory` 新增 `private int _nextItemInstanceId`，每次 `PlacedItem` 创建时 `_nextItemInstanceId++` 并赋值
- 加载完成后 `_nextItemInstanceId = max(所有已恢复 instanceId) + 1`

**物品查找机制**:

当前代码中物品存储为**直接 `ItemData` ScriptableObject 引用**（`PlacedItem.itemData` 是 `ItemData` 类型），无法用 JSON 序列化。解决方案：
1. **新建 `ItemCatalog`** ScriptableObject：编辑器自动扫描所有 `ItemData`，运行时提供 `Dictionary<string, ItemData>` O(1) 查找
2. 存盘时：`p.itemData.itemName` → 字符串
3. 读档时：`ItemCatalog.Find(itemName)` → `ItemData` 引用

**ItemCatalog 唯一性校验**:
```csharp
public void Build()
{
    var allItems = Resources.LoadAll<ItemData>("Items");
    var seen = new Dictionary<string, ItemData>();
    foreach (var item in allItems)
    {
        if (seen.TryGetValue(item.itemName, out var existing))
        {
            Debug.LogError($"[ItemCatalog] itemName 重复！" +
                $"{item.name} 和 {existing.name} 都用了 itemName='{item.itemName}'。" +
                $"加载存档时会返回错误的物品。请修改其中一个的 itemName。");
            continue; // 跳过重复项，不加入字典
        }
        seen[item.itemName] = item;
    }
    _lookup = seen;
}
```

## 7. 保存 / 加载流程

### 7.1 保存流程

```
1. SaveLoadManager 收到保存请求 (手动 / 自动)
   → 如果 _serializingLock：
       设置 _pendingSave = true（合并策略：完成当前保存后检查并执行一次）
       return（不丢弃请求，但也不堆积）
   → 设置 _serializingLock = true
   → 如果 _isWorldGenerating，排队等待 WorldGenCompleted
       │
2. 主线程同步采集所有系统数据 (构建 SaveData 对象树)
   ├─ SurvivalSystem.GetSaveData()          <1ms
   ├─ SurvivalXPSystem.GetSaveData()        <1ms
   ├─ StaminaSystem.GetSaveData()           <1ms
   ├─ Inventory.GetSaveData()               <1ms
   ├─ BuildModeController.GetSaveData()     <1ms
   ├─ WorldItem.GetSaveData()               <1ms (FindObjectsOfType)
   ├─ ContainerRegistry.GetSaveData()       <1ms
   ├─ PowerGrid.GetSaveData()               <1ms (FindObjectsOfType)
   ├─ AIBot.GetSaveData()                   <1ms (FindObjectsOfType)
   ├─ Vehicle.GetSaveData()                 <1ms (FindObjectsOfType)
   ├─ Zombie.GetSaveData(radius)            <1ms (FindObjectsOfType, radius=30m)
   ├─ TimeManager.GetSaveData()             <1ms
   ├─ WeatherManager.GetSaveData()          <1ms
   └─ FactionSystem.GetSaveData()           <1ms
       │  合计采集: <5ms
       │
       ⚠️ 关键步骤：深拷贝
       │  所有 SaveData 子结构实现 ICloneable，逐字段手动拷贝
       │  目的：切断主线程对象与后台线程间的引用共享
       │  性能：手动逐字段拷贝 ~1-2ms（vs Newtonsoft.Json 序列化+反序列化 ~50ms）
       │  ⚠️ 不可用 Newtonsoft.Json 在主线程做深拷贝——500KB 数据 ~50ms 会造成 3 帧卡顿
       ↓
3. Task.Run 后台线程（持有 SaveData 独立副本）
   ├─ Newtonsoft.Json 序列化               ~50ms
   ├─ 写入 save.json.tmp
   ├─ File.Move(save.json → save.json.backup)（原子重命名）
   └─ File.Move(save.json.tmp → save.json)  （原子重命名）
       │
4. 主线程完成收尾
   ├─ 写 meta.json（纯文本，<1ms）
   ├─ 异步截图：RenderTexture.GetTemporary → AsyncGPUReadback.Request → 回调中编码 PNG
   │   主线程不等待，截图完成后再写 thumb.png
   ├─ _serializingLock = false
   └─ 检查 _pendingSave：若为 true，清除标记并重新执行步骤1（合并保存）
       │
5. EventBus.Publish(SaveCompleted { slotIndex, success: true })
```

**深拷贝方案**:

```csharp
// ❌ 不可用：Newtonsoft.Json 序列化+反序列化在主线程 ~50ms，造成 3 帧卡顿
// ❌ 也不可用：JsonUtility 不支持 Dictionary，会丢数据

// ✅ 方案：所有 [Serializable] 数据类实现 ICloneable，逐字段手动拷贝
public class PlayerSaveData : ICloneable
{
    public object Clone()
    {
        return new PlayerSaveData
        {
            posX = this.posX, posY = this.posY, posZ = this.posZ,
            rotY = this.rotY,
            health = this.health, hunger = this.hunger, // ...
            // List/Dictionary 逐元素拷贝
            activeStates = new List<string>(this.activeStates),
            medCooldownsRemaining = new Dictionary<string, float>(this.medCooldownsRemaining),
            // ...
        };
    }
}
// 优势：纯栈上字段拷贝，~1-2ms 完成全部 SaveData 深拷贝
// 代价：每个新字段需手动添加到 Clone()，但此开销极小
```

**并发保存合并策略**:

```csharp
// 保存请求入口
public void RequestSave(int slotIndex, bool isAutoSave)
{
    if (_serializingLock)
    {
        _pendingSave = true;  // 合并而非丢弃：当前保存完成后自动触发一次
        return;
    }
    StartCoroutine(SaveCoroutine(slotIndex, isAutoSave));
}

// 保存完成后
private IEnumerator SaveCoroutine(int slotIndex, bool isAutoSave)
{
    _serializingLock = true;
    // ... 采集、深拷贝、后台线程写入 ...
    yield return new WaitUntil(() => !_backgroundTaskRunning);
    _serializingLock = false;

    if (_pendingSave)
    {
        _pendingSave = false;
        // 立即执行一次合并保存（中间累积的变化全部在此次保存）
        StartCoroutine(SaveCoroutine(slotIndex, true));
    }
}
```

### 7.2 加载流程

```
-1. 卸载当前世界（如果已存在）  ← 关键：ESC菜单"加载另一个存档"时场景中已有旧实体
   → 销毁所有持久化实体：
       - 遍历 PersistentGUIDRegistry 中的所有 GO → Destroy()
       - FindObjectsOfType<PlacedStructure>() → Destroy()
       - FindObjectsOfType<WorldItem>() → Destroy()
       - FindObjectsOfType<VehicleController>() → Destroy()
       - FindObjectsOfType<AIBot>() → Destroy()
       - FindObjectsOfType<ZombieController>() → Destroy()
       - 销毁所有 PowerSource/PowerTerminal/PowerConsumer
   → PersistentGUIDRegistry.Clear()
   → ContainerRegistry 重置（清空 _records 和 _chunkIndex）
   → 重置 WorldItem.nextInstanceId、Inventory._nextItemInstanceId
   → 确认场景干净后进入 Phase 0
       │
0. 全局冻结
   → 设置 isLoading = true 标记
   → Time.timeScale = 0          ← 暂停所有游戏逻辑
   → SaveLoadManager 注册高优先级 InputRouter 绑定，消费所有按键（屏蔽输入）
   → 显示遮罩 + "加载中..." 提示
       │
       ⚠️ 注意：项目当前无 GameState 全局状态机、无 InputLockManager。
       Phase 1 将在 SaveLoadManager 中实现简易的 isloading + InputRouter 拦截方案。
       Phase 5 可升级为完整 GameState 系统。
       │
1. 读取 save.json → JsonConvert.DeserializeObject<SaveData>
   版本校验 → 如需迁移，运行 Migration 链条
       │
2. Phase 1: 设置 WorldGenerator.seed
   → 触发 WorldGenerator 管线 (Seed→CityLayout→ModuleGrid→Mesh)
   → ⏳ 等待 WorldGenCompleted 事件 ⚠️ 关键同步点
       │
3. Phase 2: 放置玩家
   → player.transform.SetPositionAndRotation(savedPos, savedRot)
       │
4. Phase 3: 恢复玩家内部状态
   → SurvivalSystem.RestoreFromSave(playerData)
   → SurvivalXPSystem.RestoreFromSave(playerData)
   → StaminaSystem.RestoreFromSave(playerData)
   → Inventory.RestoreFromSave(inventoryData)
       │
5. Phase 4: 恢复世界实体（批量 Instantiate）
   → foreach building: Instantiate(prefab) → PersistentGUID.Initialize(guid) → 注册
   → foreach groundItem: Instantiate WorldItem → 设置 itemData + count + instanceId
   → foreach container: ContainerRegistry.Restore(record)
   ⚠️ 此阶段必须全部完成，所有 GUID 已注册 → 发 WorldEntitiesRestored 事件
       │
6. Phase 5: 恢复系统实体（依赖 Phase 4 注册的 GUID）
   → Zombie: 恢复范围内僵尸（通过 GUID 注册）
   → PowerGrid: 按 Terminal 的 connectedGuids[] 重建连线
   → AIBot: Instantiate → Initialize(guid) → 恢复状态
   → ProductionDevice: 同上
   → Vehicle: 同上
       │
7. Phase 6: 恢复全局状态
   → TimeManager: 设置 _totalGameDays
   → WeatherManager: 恢复 currentWeather + stormCooldown + consecutiveRainCount
   → ChemicalResearch: 恢复 completedIds
   → FactionSystem: 应用 factionDeltas
       │
8. Phase 7: 解冻 + 刷新
   → 所有实体注册完毕、引用完整
   → isloading = false
   → Time.timeScale = 1
   → 移除 InputRouter 高优先级拦截
   → 隐藏遮罩
   → 刷新所有 UI + EventBus.Publish(LoadCompleted)
```

**加载安全三原则**:

| 原则 | 实现 | 为什么 |
|------|------|--------|
| **时间冻结** | `Time.timeScale = 0` | 僵尸不会在加载间隙移动，生存值不会衰减 |
| **输入屏蔽** | `InputRouter.BindKey` 高优先级拦截所有键鼠输入 | 玩家无法在实体未就绪时操作，防止空引用 |
| **isloading 守卫** | 各系统 Update 开头检查 `SaveLoadManager.isLoading` | 双重保险，防止任何游戏逻辑在加载期间运行 |

**三个关键同步点**:

| 同步点 | 等待什么 | 为什么必须等 |
|--------|---------|-------------|
| Phase 2 之前 | `WorldGenCompleted` 事件 | 建筑/容器都在生成的世界里，没生成完没法放 |
| Phase 5 之前 | Phase 4 全部 Instantiate 完成 + 所有 GUID 已注册 | PowerTerminal 的 connectedGuids 指向的建筑必须在 Registry 里 |
| Phase 7 | Phase 0~6 全部完成 | 解冻前确保所有实体引用完整，否则空引用崩溃 |

### 8.1 防洗档机制

**问题**：玩家可在被僵尸围攻时 Alt+F4 强退 → 读档 → 僵尸清空 → 威胁消失。

**对策**：

| 机制 | 实现 |
|------|------|
| **睡觉存档前置检查** | 睡觉前检查 25m 内无僵尸 → 有则"附近有僵尸，无法入睡" |
| **退出保存 30s 冷却** | 上次保存后 30s 内不能再次手动保存（堵漏洞窗口） |
| **定期自动保存** | 每 10 分钟自动保存（含范围内僵尸）。即使玩家用任务管理器杀进程，最多损失 10 分钟进度。`OnApplicationQuit` 不可靠——任务管理器强杀不会触发。 |
| **加载时恢复僵尸** | §6.5 的 `ZombieSaveData` + §7.2 Phase 5 恢复 — 玩家读档后僵尸还在原地 |

**自动保存的脏标记策略**: SaveLoadManager 订阅 `StructurePlacedEvent` / `StructureDeconstructedEvent` / `ContainerOpenedEvent`。有脏数据且满足时间条件时触发。

**性能原则**: 实体数量 < 50 可用 `FindObjectsOfType<T>()`；实体数量 > 50 必须在新增/销毁时维护注册表 `List<T>`，存读时遍历注册表（O(n) 遍历 vs O(n) 场景搜索，前者可控且无 GC）。建筑系统在 P3 必须建立 PlacedStructureRegistry。

## 11. 边界条件与容错

| 场景 | 处理策略 |
|------|---------|
| **存档文件损坏** | 原子写入无此问题。万一发生：加载失败 → 提示"存档损坏" → 尝试从 backup 恢复 |
| **物品 itemName 不存在** | ItemCatalog 查不到 → 跳过该物品 → `Debug.LogWarning` → 不崩溃 |
| **物品 itemName 重复** | ItemCatalog 初始化时检测，发现重复 → `Debug.LogError` + 跳过重复项。**静默无告警是最危险的** |
| **建筑 prefab 找不到** | BuildableCatalog 查不到 → 跳过该建筑 → `Debug.LogWarning` |
| **存档版本过新** | 拒绝加载 → "此存档来自更新版本的游戏，请升级游戏" |
| **存档版本过旧** | 自动运行迁移链 (v1→v2→v3→...) |
| **世界生成算法版本不匹配** | `worldGenVersion != 当前版本` → 拒绝加载 → "存档世界版本不兼容，请开始新游戏" |
| **世界生成中保存** | `_isWorldGenerating` 标记 → 设置 `_pendingSave = true`，世界生成完毕后再存 |
| **保存请求并发** | `_serializingLock` 期间收到请求 → 设置 `_pendingSave = true`（合并策略），当前保存完成后执行一次合并保存 |
| **存档槽已满** | 弹出交互式确认对话框——展示每个槽位的元数据（天数/日期/等级），玩家手动选择覆盖目标。**绝不自动删除** |
| **磁盘空间不足** | try-catch IO 异常 → 提示"磁盘空间不足无法保存" |
| **.tmp 残留文件** | `SaveLoadManager.Awake()` 扫描所有槽位 `*.tmp` → 删除 |
| **玩家死亡时自动保存** | 保存后 `SaveSlotMeta.isDead = true`。永久死亡模式：存档槽不可加载 |

## 12. 现有系统接入兼容性

以下是**已实现系统**接入存档时的注意事项：

| 系统 | 接入方式 | 注意 |
|------|---------|------|
| **生存系统** | `SurvivalSystem.GetSaveData()` → PlayerSaveData | 所有字段 private，需添加 internal 访问器或 GetSaveData 方法。`_medCooldowns` 的实时时间键需转换为游戏时间 |
| **技能XP** | `SurvivalXPSystem.GetSaveData()` → PlayerSaveData | `_skills` 是 `List<RuntimeSkill>`，不是扁平的 `Dictionary<string,int>` |
| **体力** | `StaminaSystem.GetSaveData()` → 只存 currentStamina | maxStamina 由耐力属性动态计算，不存 |
| **背包/装备** | `Inventory.GetSaveData()` 遍历 7 容器 + equipped 字典 | 物品用 itemName 字符串（需 ItemCatalog 查找），容器名用中文。equipped 是 `Dictionary<EquipSlot, ItemData>` |
| **武器系统** | 弹药数通过 `WeaponShooting._currentMag` 获取（当前为 private，需暴露）；武器槽通过 `WeaponSwitcher.ActiveSlot` 获取（已是 public） | 持枪状态已在 Inventory.equipped 中表达 |
| **建造系统** | 遍历所有 PlacedStructure（当前无注册表，需 `FindObjectsOfType` 或新建注册表） | 每个建筑需加 PersistentGUID |
| **世界容器** | `ContainerRegistry._records` 的 containerId 是 `int`（世界生成时分配），跨会话稳定 | 不需要 PersistentGUID，直接用 int containerId |
| **世界物品** | `FindObjectsOfType<WorldItem>()` | 需新增 instanceId 字段和 WorldItemManager |
| **车辆系统** | `FindObjectsOfType<VehicleController>()` | 每个 Vehicle 需要加 PersistentGUID |
| **僵尸系统** | `FindObjectsOfType<ZombieController>()` 筛选 30m 内 | 需加 PersistentGUID，加载后 5m 去重 |
| **时间系统** | `TimeManager._totalGameDays`（private，需暴露） | 可以只存这一个字段 |
| **天气系统** | `WeatherManager.CurrentWeather` + stormCooldownRemaining + consecutiveRainCount | 马尔可夫链历史窗口暂缺，Phase 5 补齐 |
| **阵营系统** | `FactionSystem` 只存运行时修改过的关系 delta | 默认关系由 FactionData ScriptableObject 决定 |
| **威胁系统** | 不存 — 威胁值由僵尸 AI 重新计算 | |
| **声音系统** | 不存 — DecibelSystem 运行时状态不持久化 | |

### 14.1 僵尸保存详细规则

| 条件 | 保存？ | 原因 |
|------|:--:|------|
| 距玩家 ≤ 30m | ✅ 全量保存 | 直接影响玩家安全，读档后必须在 |
| 距玩家 > 30m | ❌ 不保存 | 远距离僵尸由 ZoneSpawnProfile 重生成足够 |
| 僵尸已被杀死 | ❌ 不保存 | 由 ZoneSpawnProfile 决定是否重刷 |
| 僵尸处于 Stagger/倒地 | ✅ 保存 | 玩家可能正在处决中 |
| 加载时 ZoneSpawnProfile 生成的僵尸与存档僵尸重叠（≤5m） | 删除生成的那个，保留存档的 | 避免"双僵尸"重叠站立 |

**性能估算**: 通常 30m 半径内最多 15~30 只僵尸（就算尸潮）。每只 ZombieSaveData ~150 bytes → 总计 ~2.2~4.5KB，对存档大小影响可忽略。
