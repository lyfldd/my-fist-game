### 1.2 核心设计决策

| 决策 | 说明 |
|------|------|
| **纯 JSON** | 不使用 SQLite，存档 200-500KB，记事本即可调试 |
| **Newtonsoft.Json** | 支持 Dictionary、List 泛型，Unity 官方 UPM 包 |
| **PersistentGUID** | UUID 字符串标识每个跨会话实体，替代不稳定的 InstanceID |
| **worldSeed 增量保存** | 世界由 seed 程序生成，只存玩家改动部分 |
| **itemName 字符串键** | 不存 Unity ScriptableObject 引用，通过 ItemCatalog 查表 |
| **后台线程写盘** | 主线程采集+深拷贝 → Task.Run 后台序列化/写盘 |
| **ICloneable 深拷贝** | 手动逐字段拷贝 ~1-2ms，避免 JsonUtility/Newtonsoft 在主线程的 ~50ms 卡顿 |
| **原子写入** | .tmp → backup rename → 正式档 rename，操作系统保证原子性 |

## 3. 原子写入与崩溃安全

### 3.1 三步原子写入

SaveService.SaveToDisk() 在后台线程执行：

```
步骤0: Initialize() 启动时清理所有 Slot_*/*.tmp 残留文件

步骤1: File.WriteAllText(save.json.tmp, json)
       → 写临时文件，不影响正式档

步骤2: File.Delete(save.json.backup) [若存在]
       File.Move(save.json → save.json.backup)
       → 旧正式档改为备份（同分区内纯 rename，原子操作）

步骤3: File.Delete(save.json) [若存在]
       File.Move(save.json.tmp → save.json)
       → 临时文件晋升为正式档（原子 rename）
```

### 3.2 崩溃恢复矩阵

| 崩溃时刻 | save.json | save.json.backup | save.json.tmp | 恢复策略 |
|----------|-----------|------------------|---------------|---------|
| 步骤1 之前 | 完好 | 可能不存在 | 不存在 | 正式档完整，无损 |
| 步骤1 中 | 完好 | 可能不存在 | 半截文件 | 正式档完整，启动时删 .tmp |
| 步骤2 之前 | 完好 | 可能不存在 | 完整 | 正式档完整，启动时删 .tmp |
| 步骤2 中 | 完好或丢失 | 可能半截 | 完整 | rename 原子，要么旧 backup 完整要么新 backup 半截，正式档不受影响 |
| 步骤3 之前 | 完好 | 上版完好 | 完整 | 正式档完整，启动时删 .tmp |
| 步骤3 中 | 半截或丢失 | 上版完好 | 完整 | rename 原子，正式档要么为新要么为旧 |
| 步骤3 之后 | 新档完好 | 上版完好 | 不存在 | 新档就位，完美 |

### 3.3 加载时的 backup 恢复

```csharp
public static SaveData Load(int slotIndex)
{
    var savePath = Path.Combine(slotDir, "save.json");
    if (!File.Exists(savePath))
    {
        var backupPath = Path.Combine(slotDir, "save.json.backup");
        if (File.Exists(backupPath))
        {
            Debug.LogWarning($"save.json 不存在，从 backup 恢复 slot={slotIndex}");
            File.Move(backupPath, savePath);  // backup → 正式档
        }
        else return null;  // 两者都不存在 = 空槽位
    }
    // 正常读盘...
}
```

### 3.4 退出时同步保存

`OnDestroy()` 调用 `AutoSaveOnQuit()`：
- 检查 `IsLoading` 和 `PlayerRegistry.Transform == null`（主菜单无玩家不保存）
- 使用 `Task.Wait()` 同步等待（退出时不能异步，Unity 会立即卸载）
- 同样的采集 → 深拷贝 → 后台 Task → Wait 流程

## 4. 后台线程序列化

### 4.1 主线程数据采集

`SaveCoroutine()` 在主线程同步采集所有系统数据，构建 `SaveData` 对象树：

```csharp
var saveData = new SaveData { ... };
saveData.player      = CollectPlayerData();      // SurvivalSystem + XP + Stamina
saveData.inventory   = CollectInventoryData();    // 7容器 + 装备 + 武器弹药
saveData.world       = CollectWorldData();        // 建筑/地面物品/容器/阵营/AIBot
saveData.productions = CollectProductionData();   // 生产设备
saveData.timeWeather = CollectTimeWeatherData();  // 天数 + 天气马尔可夫链
saveData.research    = CollectResearchData();     // 化学研究
```

### 4.2 ICloneable 深拷贝

所有 SaveData 子结构实现 `ICloneable`，手动逐字段拷贝：

```csharp
var snapshot = saveData.Clone() as SaveData;  // ~1-2ms，纯栈上字段赋值
```

**为什么要手动 Clone 而不是 Newtonsoft.Json 序列化+反序列化？**

| 方案 | 耗时 | 问题 |
|------|:--:|------|
| Newtonsoft.Json 序列化+反序列化 | ~50ms | 主线程卡顿 3 帧 |
| JsonUtility | 不支持 Dictionary | 丢数据 |
| 手动 ICloneable 逐字段拷贝 | ~1-2ms | 代码稍多，但性能零开销 |

### 4.3 Task.Run 后台写盘

```csharp
var task = SaveService.SaveAsync(slotIndex, snapshot);  // 立即返回 Task

// 主线程继续执行（写 meta.json、等待 task 完成）
while (!task.IsCompleted) yield return null;  // 每帧检查一次，不阻塞

task.Wait();  // 已完成，取出异常（如果有）
```

`SaveAsync()` 内部：
```csharp
public static Task SaveAsync(int slotIndex, SaveData snapshot)
    => Task.Run(() => SaveToDisk(slotIndex, snapshot));
// SaveToDisk(): 序列化 JSON → 原子写入三步法
```

### 4.4 并发保存合并策略

```csharp
private bool _serializingLock;   // 当前正在序列化/写盘
private bool _pendingSave;       // 有排队的保存请求

public void RequestSave(int slotIndex, bool isAutoSave)
{
    if (_serializingLock) { _pendingSave = true; return; }  // 合并不丢弃
    if (IsLoading)        { return; }                        // 加载中拒绝
    if (IsWorldGenerating) { _pendingSave = true; return; }  // 世界生成完后补存
    StartCoroutine(SaveCoroutine(slotIndex, isAutoSave));
}

// SaveCoroutine 完成时:
_serializingLock = false;
if (_pendingSave) { _pendingSave = false; StartCoroutine(SaveCoroutine(...)); }
```

## 5. PersistentGUID 架构

### 5.1 为什么需要 PersistentGUID？

Unity 的 `GameObject.GetInstanceID()` 每次 Play Mode 重启都重新分配。加载存档时无法用 InstanceID 找回"上次那个建筑"。PersistentGUID 提供一个**跨会话稳定的 UUID 字符串**。

### 5.2 PersistentGUID 组件

```csharp
public class PersistentGUID : MonoBehaviour
{
    [SerializeField] private string guid;        // UUID v4（32位十六进制，无连字符）
    [SerializeField] private string entityType;  // "PlacedStructure"/"AIBot"/"Vehicle"/...

    void Awake()
    {
        if (string.IsNullOrEmpty(guid))
            guid = System.Guid.NewGuid().ToString("N");  // 首次创建时生成
    }

    void Start()
    {
        if (!_initialized)
        {
            PersistentGUIDRegistry.Register(guid, gameObject);
            _initialized = true;
        }
    }

    void OnDestroy()
    {
        PersistentGUIDRegistry.Unregister(guid);
    }

    public void Initialize(string overrideGuid, string type)
    {
        // 竞态安全：先注销 Awake/Start 可能已自动注册的旧 GUID
        if (!string.IsNullOrEmpty(guid))
            PersistentGUIDRegistry.Unregister(guid);

        guid = overrideGuid;
        entityType = type;
        PersistentGUIDRegistry.Register(guid, gameObject);
        _initialized = true;
    }
}
```

### 5.3 两阶段初始化 + 竞态安全

```
Awake()          → 生成 GUID（或保留已序列化的）
Start()          → 自动注册（给加载系统覆盖 GUID 的窗口期）
Initialize()     → 加载时显式调用：Unregister(旧) → 覆写 GUID → Register(新)
```

**竞态场景**: Unity 不保证 `Instantiate()` 后 `Start()` 在 `Initialize()` 之前执行。如果 `Start()` 先用 Awake 生成的旧 GUID 注册，`Initialize()` 再覆盖 `guid` 字段——Registry 中残留旧 GUID 脏条目。

**解决**: `Initialize()` 先 `Unregister(旧GUID)` 再 `Register(新GUID)`，无论执行顺序如何，Registry 最终都是正确的。

### 5.4 PersistentGUIDRegistry 全局查找表

```csharp
public static class PersistentGUIDRegistry
{
    private static Dictionary<string, GameObject> _lookup = new();

    public static void Register(string guid, GameObject go) => _lookup[guid] = go;
    public static void Unregister(string guid)              => _lookup.Remove(guid);
    public static GameObject Find(string guid)              => _lookup.TryGetValue(guid, out var go) ? go : null;
    public static bool Exists(string guid)                  => _lookup.ContainsKey(guid);
    public static void Clear()                              => _lookup.Clear();
    public static int Count                                 => _lookup.Count;
}
```

### 5.5 哪些实体需要 PersistentGUID？

| 需要 ✅ | 不需要 ❌ |
|---------|----------|
| PlacedStructure（建筑） | Player（唯一，无需追踪） |
| AIBot（机器人） | WorldContainer（已有稳定的 int containerId） |
| Vehicle（车辆） | 纯环境装饰物 |
| ProductionDevice（生产设备） | 地面物品（有 instanceId，Pickup 即消失） |
| PowerSource / PowerTerminal / PowerConsumer | |
| Zombie（需保存的） | |

### 5.6 加载时 GUID 覆盖流程

```
Instantiate(buildingPrefab)
    → Awake(): guid = 随机新 UUID
    → Start(): 可能已用随机 UUID 注册

Load 流程显式调用:
    go.GetComponent<PersistentGUID>().Initialize(savedGuid, "PlacedStructure")
        → Unregister(随机 UUID)
        → guid = savedGuid
        → Register(savedGuid, go)
        → 干净！
```

### 6.3 InventorySaveData — 背包/装备

采集自: `Inventory` + `WeaponSwitcher` + `WeaponShooting`

| 字段 | 说明 |
|------|------|
| containers | 7 个容器（上衣口袋/裤子口袋/腰带/胸挂/背包/头盔/防弹衣） |
| equippedItems | EquipSlot名 → EquippedItemSaveData（用 instanceId 定位物品） |
| ammoReserves | EquipSlot名 → 当前弹夹子弹数 |

```csharp

{

    public string equipSlotName;     // EquipSlot 枚举名（双保险）
    public int gridWidth, gridHeight;
    public List<SlotSaveData> slots;
}

[Serializable]
public class SlotSaveData : ICloneable
{
    public int instanceId;           // PlacedItem.instanceId（0=空格子）
    public string itemName;          // ItemData.itemName（null/空=空格子）
    public int count;
    public int gridX, gridY;
    public bool rotated;
    public bool isGhost;             // 腰带武器槽幽灵占位格
    public string ghostSourceSlot;   // 幽灵格来源
}

[Serializable]
public class EquippedItemSaveData : ICloneable

```

**instanceId 定位方案**:
- 每件 `PlacedItem` 创建时分配唯一递增 `instanceId`
- 存盘: `equippedItems["RightHand"] = { itemInstanceId: 42 }`
- 读档: 遍历所有容器找 `instanceId==42` → 装备
- **不依赖列表插入顺序**，解决 slotIndex 方案的顺序依赖问题

    public float openedGameDay;

### 6.6 TimeWeatherSaveData — 时间与天气

采集自: `TimeManager` + `WeatherManager`

| 字段 | 来源 | 说明 |
|------|------|------|
| totalGameDays | TimeManager | 浮点天数 |
| currentWeather | WeatherManager | "Sunny"/"Cloudy"/"Rain"/"Storm" |
| stormCooldownRemaining | WeatherManager | 风暴冷却剩余小时数 |

## 7. 保存流程

```
SaveLoadManager.RequestSave(slotIndex, isAutoSave)
    │
    ├─ _serializingLock? → _pendingSave = true → 合并等待
    ├─ IsLoading? → 拒绝
    ├─ IsWorldGenerating? → _pendingSave = true → 延迟
    │
    └─ StartCoroutine(SaveCoroutine(...))
        │
        1. 主线程采集所有系统数据
           ├─ CollectPlayerData()      → SurvivalSystem + XP + Stamina
           ├─ CollectInventoryData()   → Inventory + WeaponSwitcher + WeaponShooting
           ├─ CollectWorldData()       → PlacedStructureRegistry + WorldItem + ContainerRegistry
           │                            + FactionSystem + AIBot
           ├─ CollectProductionData()  → FindObjectsOfType<ProductionDevice>()
           ├─ CollectTimeWeatherData() → TimeManager + WeatherManager
           └─ CollectResearchData()    → ChemicalResearchManager
           合计: <5ms
            │
        2. ICloneable 深拷贝 → SaveData 快照 (~1-2ms)
            │
        3. Task.Run 后台线程
           ├─ JsonConvert.SerializeObject(snapshot)
           ├─ File.WriteAllText(save.json.tmp)
           ├─ File.Move(save.json → save.json.backup) [原子]
           └─ File.Move(save.json.tmp → save.json)    [原子]
            │
        4. 主线程写 meta.json (<1ms)
            │
        5. yield return null 直到 Task.IsCompleted
            │
        6. 检查 _pendingSave → 若有则重新执行
            │
        7. EventBus.Publish(SaveCompleted)
```

### 采集策略

| 实体类型 | 采集方式 | 原因 |
|----------|---------|------|
| 玩家/背包 | 直接引用（PlayerRegistry） | 唯一实体 |
| 建筑 | PlacedStructureRegistry.AllStructures | 注册表 O(n)，比 FindObjectsOfType 快 |
| 地面物品 | FindObjectsOfType<WorldItem>() | 数量可控（<200） |
| 容器 | ContainerRegistry.Instance.GetAllRecords() | 已有注册表 |
| 阵营 | FactionSystem.GetFactionDeltas() | 增量 |
| AIBot | FindObjectsOfType<AIBot>() | 数量可控（<20） |
| 生产设备 | FindObjectsOfType<ProductionDevice>() | 数量可控（<50） |

## 8. 加载流程 (Phase -1 ~ 8)

```
LoadCoroutine(slotIndex)
    │
Phase -1:  卸载当前世界
    │      ├─ DestroyEntitiesOfType<PersistentGUID>()  // 销毁所有持久化实体
    │      ├─ PersistentGUIDRegistry.Clear()
    │      └─ 重置 ID 分配器
    │
Phase 0:   全局冻结
    │      ├─ IsLoading = true
    │      ├─ Time.timeScale = 0        // 暂停所有游戏逻辑
    │      ├─ InputRouter 高优先级拦截所有键鼠输入（F1-F12, Esc, E, Tab, I, B, T, P, V, 鼠标0/1/2）
    │      └─ EventBus.Publish(GameLoadStarted)
    │
Phase 1:   读取存档
    │      ├─ SaveService.Load(slotIndex) → SaveData
    │      ├─ 版本校验（worldGenVersion 不匹配 → 拒绝加载）
    │      └─ null 检查（失败 → Unfreeze + LoadCompleted(false)）
    │
Phase 2:   设置世界种子 → 触发 WorldGenerator
    │      [TODO] WorldGenerator.SetSeed(saveData.worldSeed)
    │
Phase 3:   放置玩家
    │      ├─ player.SetPositionAndRotation(savedPos, savedRot)
    │      ├─ RestorePlayerState()  → SurvivalSystem + XP + Stamina
    │      └─ RestoreInventory()    → Inventory + equipped + ammoReserves
    │
Phase 4:   恢复世界实体
    │      ├─ 恢复建筑: foreach buildings → BuildModeController.PlaceStructureDirect()
    │      │   → PersistentGUID.Initialize(savedGuid) → 覆盖 GUID
    │      ├─ 恢复地面物品: new GameObject + WorldItem 组件 → instanceId
    │      │   → 重置 WorldItemManager 计数器
    │      ├─ 恢复容器: ContainerRegistry.RestoreRecord()
    │      ├─ 恢复阵营: FactionSystem.ApplyDeltas()
    │      └─ EventBus.Publish(WorldEntitiesRestored)
    │
Phase 5:   恢复 AI 机器人（独立实体路径）
    │      ├─ foreach aiBots → PlaceStructureDirect() → GUID 覆盖
    │      └─ bot.RestoreFromSave() 内部状态
    │
Phase 6:   恢复系统实体状态
    │      ├─ RestoreProductionDevices()  // 按 GUID 查找 + 两阶段恢复（内部状态 + 设备链接）
    │      └─ [TODO] 电力网/车辆/僵尸
    │
Phase 7:   恢复全局状态
    │      ├─ RestoreTimeWeather() → TimeManager + WeatherManager
    │      └─ RestoreResearch()    → ChemicalResearchManager
    │
Phase 8:   解冻
           ├─ IsLoading = false
           ├─ Time.timeScale = 1
           ├─ InputRouter.UnbindAll(this)
           └─ EventBus.Publish(LoadCompleted)
```

### 加载安全三原则

| 原则 | 实现 |
|------|------|
| **时间冻结** | `Time.timeScale = 0` — 僵尸不移动，生存值不衰减 |
| **输入屏蔽** | InputRouter 高优先级拦截全部键鼠 — 玩家无法在实体未就绪时操作 |
| **isLoading 守卫** | 各系统 Update 开头检查 `SaveLoadManager.IsLoading` |

## 12. 版本迁移框架

`SaveMigration.cs`：

```csharp
public static class SaveMigration
{
    public const int CURRENT_VERSION = 1;

    // 版本号 → 迁移函数（vN → vN+1）
    private static Dictionary<int, Func<SaveData, SaveData>> _migrationChain = new()
    {
        // { 1, MigrateV1ToV2 },  // 示例（Phase 5 实际使用时注册）
    };

    public static SaveData Migrate(SaveData data)
    {
        while (data.version < CURRENT_VERSION)
        {
            if (!_migrationChain.TryGetValue(data.version, out var migrateFunc))
                return null;  // 缺少迁移函数
            data = migrateFunc(data);
            data.version++;
        }
        return data;
    }
}
```

**迁移是纯函数**: SaveData v(N) → SaveData v(N+1)。加载时检测 `SaveData.version`，自动串行运行迁移链。

## 14. 边界条件与容错

| 场景 | 处理策略 |
|------|---------|
| **存档文件损坏** | 原子写入避免。万一发生：尝试从 backup 恢复 |
| **物品 itemName 不存在** | ItemCatalog 查不到 → 跳过该物品 → LogWarning → 不崩溃 |
| **物品 itemName 重复** | ItemCatalog.Build() 检测 → LogError + 跳过重复项 |
| **建筑 prefab 找不到** | BuildableCatalog 查不到 → 跳过该建筑 → LogWarning |
| **存档版本过新/过旧** | 过新拒绝+提示；过旧自动迁移 |
| **worldGenVersion 不匹配** | 拒绝加载 → "世界版本不兼容，请开始新游戏" |
| **世界生成中保存** | 设置 `_pendingSave`，生成完毕后补存 |
| **并发保存** | `_pendingSave` 合并策略 |
| **退出保存** | 同步 Task.Wait()，检查非主菜单（PlayerRegistry 非空） |
| **.tmp 残留** | Initialize() 扫描删除所有 `*.tmp` |
| **死亡存档** | isDead=true，永久死亡模式不可加载 |
| **磁盘不足** | try-catch IO 异常 → 提示用户 |
| **加载时输入** | InputRouter 高优先级消费全部按键 |
