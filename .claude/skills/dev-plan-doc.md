# Game Development Plan Document Manager

## 项目前提

- **项目**: 俯视角 3D 末日生存沙盒 Unity 游戏
- **架构原则**: 低耦合、高内聚、预留扩展接口、优先 EventBus 通信解耦
- **开发模式**: 模块拆分→分阶段分步实现，先基础后进阶，循序渐进迭代

## 核心执行规则

### 最小记录单位

以「单个游戏模块下的单个开发步骤」作为计划书最细记录单元。

例: 建造系统-步骤1 基础搭建、建造系统-步骤2 承重机制、战斗系统-步骤1 基础攻击

### 三段式固定存档格式（强制留存三份）

#### ① 前置初始计划【动工前定稿】

- **阶段**: 开发步骤正式开始之前
- **内容**: 双方讨论敲定最终执行方案、实现思路、逻辑流程、关联系统对接方式、代码架构思路、预期实现效果、开发难点预判、依赖资源与接口规划
- **用途**: 作为本步骤原始执行纲领，严格按此方案动工

#### ② 后置定稿计划【开发完成后修订版】

- **阶段**: 该开发步骤全部开发调试完毕
- **内容**: 结合实际开发过程，修正落地细节、调整逻辑流程、补充实际代码实现方案、修改原不合理设计、新增临时适配逻辑、补齐 BUG 处理方案、优化调整内容、实际对接其他模块真实逻辑
- **用途**: 作为该步骤最终落地正式定稿，录入项目正式开发文档

#### ③ 前后差异对比分析表

- **阶段**: 后置计划完成后同步生成
- **内容**: 逐条对比前置与后置计划

| 对比维度 | 内容 |
|----------|------|
| 原计划设计 | 前置计划中列出的设计内容 |
| 实际落地修改 | 后置计划中变更的内容 |
| 变更原因 | 技术限制 / 逻辑冲突 / 玩法调整 / 性能问题 / 跨模块兼容问题 |
| 优化增补 | 开发过程中新增的优化项 |
| 删减废弃 | 计划有但最终未实现的内容 |
| 对后续影响 | 对后续步骤、关联模块造成的影响与适配方案 |

### 计划书录入统一规范

每条步骤记录必须标注:
- 所属模块 + 开发阶段序号 + 步骤名称 + 开发起止状态
- 示例: `建造系统 - 第1阶段 - 基础建造UI与放置功能`

### 内容必填板块（前后两份计划统一结构）

1. **本步骤核心开发目标**
2. **关联联动模块** — 对接哪些系统、数据互通方式
3. **整体实现流程拆解**
4. **核心代码/架构设计思路**
5. **解耦与可扩展预留设计**
6. **资源、配置、数据表规划**
7. **预期功能边界** — 做什么、不做什么
8. **调试与测试验收标准**

### 联动开发规则

| 阶段 | 规则 |
|------|------|
| 动工前 | 对齐跨模块关联关系，确认接口、事件、数据字段无冲突，再写入前置计划 |
| 开发中 | 不随意改动已定前置方案，如需临时调整先记录变更点 |
| 完成后 | 统一汇总所有改动，整理为后置定稿，严禁前后文档内容混乱 |

### 适用场景

战斗系统、背包系统、生存数值系统、僵尸 AI、建造系统、载具系统、物资掉落、地图生成、UI 系统、物品制作等所有游戏功能模块。

## 使用指令

用户告知: **所属模块 + 当前第几个开发步骤 + 步骤功能名称**

执行流程:
1. 输出前置动工初始计划，写入开发计划书存档
2. 开发完成后，提交实际开发内容，生成后置最终定稿计划
3. 自动整理生成前后计划差异对比文档，统一归档进项目开发计划书

---

## 约束要求

- 所有计划内容贴合 Unity C# 开发逻辑，贴合项目现有架构，不脱离现有模块设计
- 全程注重解耦设计、预留扩展接口，所有调整内容优先保证不破坏前期已有系统
- 对比分析客观写实，区分「合理优化调整」和「设计失误整改」两类变更原因
- 文档格式整洁结构化，可直接复制存入正式开发计划书文档
